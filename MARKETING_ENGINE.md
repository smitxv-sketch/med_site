# Архитектура Маркетингового Движка (Contextual Marketing Engine)

## 1. Критическая оценка изначальной задумки
Действительно, привязка **только** к UTM-меткам — это хороший старт, но с точки зрения Enterprise-маркетинга (на уровне таких систем как Dynamic Yield, Optimizely или Mutiny) этого недостаточно. 

**Почему жесткая привязка к UTM — это "фигня" (в хорошем смысле, как точка роста):**
1. Пользователь может прийти без UTM (стереть из ссылки, прямой заход, органика).
2. Маркетологу нужно управлять ожиданиями пользователя *в контексте его ситуации*, а не только источника трафика. 
3. Настройки UI — это лишь "краски". Настоящий маркетинг — это смена **порядка блоков, тона коммуникации (Tone of Voice) и смысловых акцентов**.

**Вывод:** Нам нужна архитектура **"Правил (Rules)"**, где `UI Preset` — это лишь следствие определенного `Trigger` (события).

---

## 2. Разделение логики: Trigger -> Action (Preset)

Центр Управления Маркетингом в Strapi (и движок на фронтенде) должен быть разделен на 3 слоя:

1. **Библиотека Дизайн-токенов (Генератор)**
   Определяет базовые стили: цвета (brandHue), скругления (appRadius), стили теней.
2. **Библиотека Пресетов (Обертки)**
   Сохраненные комбинации токенов + включенные/выключенные секции. 
   *Пример: "Пресет Премиум: черные тона, острые углы, шрифт Playfair, скрыт блок ОМС".*
3. **Библиотека Правил (Rules Engine)**
   Логика: `IF [Conditions] THEN [Apply Preset]`.

## 3. Мощные мировые практики: Что реально нужно Маркетологу?
Маркетолог часто не знает, что *так можно было*. Если дать ему "библиотеку сценариев" из коробки, это вызовет вау-эффект.

### Сценарий 1: "Ночная Смена" (Time-based urgency)
* **Триггер:** Локальное время пользователя с 22:00 до 07:00.
* **Экшен:** Применить `Emergency Preset`.
* **Что происходит:** Интерфейс переключается в темную или спокойную тему. Блоки "Косметология" и "Профилактика" уезжают вниз. На первом экране появляется огромная карточка: "Травмпункт 24/7. Вызов скорой помощи. Дежурный врач".
* **Польза:** Резкий рост конверсии на высокомаржинальные срочные услуги ночью.

### Сценарий 2: "Детский трафик из TikTok/Inst" (UTM-based + Referrer)
* **Триггер:** `utm_source=tiktok` ИЛИ `utm_campaign=kids_promo`.
* **Экшен:** `Kids Friendly Preset`.
* **Что происходит:** Формы кнопок становятся максимально круглыми (`rounded-full`), цвета — яркие (оранжевый/айвори), шрифты более дружелюбные. Главный баннер: "Сдаем анализы без слез". Блок "Взрослая клиника" сворачивается.
* **Польза:** Снижение показателя отказов (Bounce Rate) у мам, перешедших по рекламе. Строгое попадание в ожидание.

### Сценарий 3: "Охота за Премиумом" (Search Intent / Query)
* **Триггер:** `utm_term` содержит "дорого", "люкс" ИЛИ переход с элитного портала.
* **Экшен:** `Premium Aesthetics Preset`.
* **Что происходит:** Убираются все плашки "Акции", "Скидки" и перечеркнутые цены. Радиус закруглений становится `0` (строгий, газетный стиль). Цветовая гамма — черный/золотой/беж. Акцент на приватность, VIP-палаты и чекапы за 500k руб.
* **Польза:** Обеспечение консистентности для VIP-клиентов. Дешевый дизайн отпугивает премиум-аудиторию.

### Сценарий 4: "Дожим возвращенцев" (Behavioral Retargeting)
* **Триггер:** Сессия пользователя > 1 И прошлый визит содержал просмотр "Имплантация".
* **Экшен:** `Push Conversion Preset`.
* **Что происходит:** Включается виджет `UrgencyLevel = high`. На главном экране вместо абстрактной клиники — плашка "Только 2 дня: Бесплатная консультация хирурга-имплантолога". 
* **Польза:** Работа с воронкой на уровне интерфейса всего сайта.

## 4. Как это связать с Backend (Strapi JSON Structure)

Чтобы всё работало как часы, в Strapi создается коллекция `Marketing Rules`. 
Она отдает на фронтенд массив JSON-правил (оцениваются сверху вниз, как в Firewall):

```json
[
  {
    "id": "rule_night_emergency",
    "name": "Ночная срочность",
    "priority": 100,
    "conditions": {
      "operator": "AND",
      "rules": [
        { "field": "time", "operator": "greater_than", "value": "22:00" },
        { "field": "time", "operator": "less_than", "value": "07:00" }
      ]
    },
    "actions": {
      "applyPreset": "emergency_preset_id",
      "overrideBlocks": {
        "heroSection": { "variant": "B", "title": "Травмпункт 24/7" }
      }
    }
  },
  {
    "id": "rule_kids_tiktok",
    "priority": 90,
    "conditions": {
      "operator": "OR",
      "rules": [
        { "field": "urlParam", "key": "utm_campaign", "operator": "equals", "value": "kids" }
      ]
    },
    "actions": {
      "applyPreset": "kids_theme_id"
    }
  }
]
```

## 5. Как это работает в браузере (Frontend)
1. Пользователь заходит на сайт.
2. Скачивается статика (Сайт по умолчанию, пресет `Universal`).
3. На этапе гидратации (React/Zustand) запускается `Rule Evaluator Function`.
4. Функция смотрит время, читает URL, проверяет localStorage.
5. Если правило срабатывает — Zustand мгновенно переключает `UISettingsStore`, и сайт (без перезагрузки и морганий) трансформируется под пользователя.

**Почему это продает ваш продукт?**
Потому что вы продаете клиникам не "красивый сайтик", а **инструмент генерации выручки**, который понимает психологию пациента. Это уровень, недоступный обычным студиям веб-разработки.
