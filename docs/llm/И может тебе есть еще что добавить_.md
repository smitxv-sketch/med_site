# И может тебе есть еще что добавить?

Да — три момента, которые я ещё не покрыл полностью, но которые на практике часто стреляют.

## 1. Schema versioning как first-class concern

Каждый раз когда ты меняешь Zod-схему фазы — это breaking change для всего downstream, включая Prisma-записи, repair prompts, canary comparisons и cost tracking. Я бы заложил `schemaVersion` с самого начала:

```ts
interface PhaseDefinition<T> {
  phase: PhaseKey;
  version: number; // bump on EVERY schema change
  schema: z.ZodType<T>;
  // ...
}
```

В Prisma-записи `PhaseAttempt` хранится `schemaHash` — это sha256 от JSON Schema строки. Это позволяет в observability группировать `schema_pass_rate` по версии схемы, а не по дате, и видеть когда ты ввёл новое required поле и сломал старую модель.

**Практичное правило:** при bump схемы всегда прогоняй golden corpus (§3 из прошлого сообщения) на новой версии **до** деплоя. Иначе `schema_pass_rate` упадёт и ты не поймёшь — это провайдер деградировал или ты сам сломал.

***

## 2. Cost budgeting per car, не per request

У тебя фраза `cost_per_car` в метриках — это правильно, но нужно явно управлять этим бюджетом. Типичная проблема: одна фаза на одном сложном авто делает 3 retry × большая схема = стоимость x5 от среднего. Без hard cap на run-уровне это молча утекает:

```ts
// src/llm/budget.ts
interface RunBudget {
  maxUsdPerCar: number;       // e.g., $0.15
  maxTokensPerPhase: number;  // hard cap on output tokens
  maxAttemptsTotal: number;   // across ALL phases in one run
}

class BudgetGuard {
  private spentUsd = 0;
  private attemptsUsed = 0;

  charge(costUsd: number, attempts: number) {
    this.spentUsd += costUsd;
    this.attemptsUsed += attempts;
    if (this.spentUsd > this.budget.maxUsdPerCar) {
      throw new BudgetExceededError("usd", this.spentUsd, this.budget.maxUsdPerCar);
    }
  }
}
```

Это также помогает отлавливать petty loops — когда repair → fail → repair → fail гоняет по кругу и ты платишь за нерабочий контент.

***

## 3. "Dry run" режим для prompt changes

Ты скорее всего будешь регулярно итерировать промпты фаз. Самая частая ошибка — деплоить новый промпт сразу на production трафик, потому что "ну там же просто формулировка изменилась". На практике даже мелкое изменение формулировки меняет `schema_pass_rate` на reasoning-моделях.

Я бы добавил `dryRun: boolean` флаг на уровне `PhaseDefinition`, который:

1. Прогоняет фазу нормально, пишет в Prisma — но **не** кладёт результат в основной enrichment pipeline
2. Сравнивает `normalizedJson` с golden baseline через semantic diff
3. Пишет event `dry_run_deviation` с полем `deviationScore`

Это позволяет тестировать новый промпт параллельно с production без риска положить плохой результат в БД.

## Что ещё стоит держать в голове

Вот короткий список неочевидных деталей, которые я бы занёс в `ARCHITECTURE.md` проекта прямо сейчас, пока не забылось:


| Аспект | Правило |
| :-- | :-- |
| DeepSeek reasoner multi-turn | Никогда не класть `reasoning_content` обратно в messages — провайдер вернёт 400 [^1] |
| OpenRouter `require_parameters` | Без него роутер может тихо деградировать json_schema → json_object [^2][^3] |
| Claude prefill `{"` | Не работает на Claude 4.5+ structured outputs, ломает API [^4] |
| Gemini ключи | До ноября 2025 сортировал ключи алфавитно, сейчас сохраняет порядок схемы [^5] |
| Repair context | Никогда не передавать оригинальный system prompt в repair — там может быть бизнес-логика/PII [^6] |
| Zod `.describe()` | Добавлять на каждое поле — это попадает в JSON Schema `description` и реально помогает моделям [^7] |
| `finish_reason === "length"` | Не парсить, сразу retry с x1.5 max_tokens или split schema [^8] |
| Schema complexity score | Считать `flatFieldCount + nestingDepth * 3` и логировать — коррелирует с truncation rate [^9] |
