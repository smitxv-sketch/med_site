## Статические vs. Динамические страницы в Strapi + Next.js

В Strapi есть два типа контента, которые напрямую соответствуют твоей задаче [^1]:


| Тип | Strapi | Next.js | Пример |
| :-- | :-- | :-- | :-- |
| **Single Type** | Одна запись | Статическая страница (`/`) | Главная, Контакты |
| **Collection Type** | Множество записей | Динамический роут `[slug].tsx` | Врачи, Услуги, Новости |
| **Page Builder** | Collection Type + Dynamic Zone | Тот же `[slug].tsx` | Любая кастомная страница |

Для тысяч карточек (врачи, услуги) ты создаёшь один файл `app/doctors/[slug]/page.tsx`, Next.js генерирует все страницы через `generateStaticParams()` — это SSG + ISR [^2]. Никакой дублированной вёрстки.

## Dynamic Zones — это и есть твой Page Builder

Ты не изобретаешь велосипед, но добавляешь к нему мотор. Вот как выглядит стандартная архитектура Tilda-подобного редактора на Strapi [^3]:

**В Strapi:**

```
Collection Type: Page
  - title: string
  - slug: string (UID)
  - content: Dynamic Zone
      ├── widget.hero
      ├── widget.doctors-list
      ├── widget.reviews
      ├── widget.promo-banner
      └── widget.faq
```

Каждый `widget.*` — это **Component** в Strapi (аналог Custom Post Type в WordPress) [^4]. У него свои поля: `variant` (A/B/C/D), `itemsCount`, `animationStyle, мобильный вариант и т.д.

**На фронте (Next.js):**

```tsx
// app/[slug]/page.tsx
export default function Page({ data }) {
  return data.content.map((block) => {
    switch (block.__component) {
      case 'widget.doctors-list': return <DoctorsList key={block.id} {...block} />;
      case 'widget.reviews':      return <Reviews key={block.id} {...block} />;
      // ...
    }
  });
}
```

Это паттерн называется **Block Renderer** или **Component Switch** [^1] — один универсальный рендерер, который диспатчит компоненты по типу блока.

## Деплой без 10-минутного ребилда — ISR + Webhook

Твоя идея «нажал Сохранить → задеплоил» реализуется через **On-Demand ISR**, а не через полный ребилд [^5][^6]:

1. Контент-менеджер меняет страницу в Strapi и нажимает **Publish**
2. Strapi отправляет **Webhook** на твой Next.js API endpoint: `POST /api/revalidate?token=xxx`
3. Next.js вызывает `revalidatePath('/about')` или `revalidateTag('doctors')` [^7]
4. Конкретная страница пересобирается **в фоне** за секунды — без ребилда всего сайта
```ts
// app/api/revalidate/route.ts
import { revalidateTag } from 'next/cache';
export async function POST(req: Request) {
  const { searchParams } = new URL(req.url);
  if (searchParams.get('token') !== process.env.REVALIDATE_TOKEN) 
    return Response.json({ error: 'Unauthorized' }, { status: 401 });
  
  const body = await req.json();
  revalidateTag(body.model); // 'doctors', 'reviews', 'pages'
  return Response.json({ revalidated: true });
}
```

Это именно то, что ты описывал — изменения сразу отражаются на сайте, но без 10 минут ожидания. На полный деплой (новый виджет в коде) ты уже используешь Coolify автодеплой через GitHub webhook [^8].

## Dev Workflow с моками — Storybook + MSW

Для разработки виджетов без Strapi используй **Storybook + Mock Service Worker (MSW)** [^9][^10]:

- Разрабатываешь виджет изолированно со своими JSON-моками
- Видишь все варианты (A/B/C/D, mobile/desktop) в одном месте
- Когда виджет готов — деплоишь через `git push`, Coolify подхватывает и собирает [^8]
- В Strapi просто добавляешь новый component в Dynamic Zone

Структура:

```
src/
  components/
    widgets/
      DoctorsList/
        index.tsx          # реальный компонент
        DoctorsList.stories.tsx  # моки для Storybook
        types.ts
```

## Архитектурные решения, которые стоит принять сразу

**1. Populate стратегия в Strapi v5** — в пятой версии нет `populate: '*'` для Dynamic Zones, только явный `on`-фрагмент [^11][^12]:

```ts
populate: {
  content: {
    on: {
      'widget.doctors-list': { populate: { doctors: true } },
      'widget.reviews': { populate: { avatar: true } },
    }
  }
}
```

**2. Page Builder vs. Fixed Templates** — для страниц типа «карточка врача» не нужен Dynamic Zone, только для редактируемых «маркетинговых» страниц. Разделяй: Collection Types для сущностей, Page Builder для лендингов.

**3. Переход существующего сайта** — работает инкрементально: подключаешь Strapi, начинаешь с одного типа контента (например, врачи), переносишь JSON → в Strapi. Остальные данные временно берёшь из JSON-файлов. Никакого Big Bang.

## Тренды, которые стоит учесть сразу

- **Strapi 5 + Next.js 15 App Router** — серверные компоненты снижают JS-бандл до нуля для статики [^13]
- **`revalidateTag`** вместо path-revalidation — более гранулярно, теги навешиваются на `fetch()` вызовы [^7]
- **Visual CMS поверх Headless** — тренд идёт в сторону Builder.io / Storyblok (встроенный visual editor), но это можно реализовать самому через свой page builder + Strapi [^14]
- **Storybook как дизайн-система** — Strapi даже написал официальный гайд по интеграции Storybook с Next.js для этого [^9]

Для твоего AI-конструктора страниц — агент генерирует JSON-структуру Dynamic Zone (массив блоков с параметрами), пишет её в Strapi через Content API, Strapi дёргает webhook, ISR пересобирает страницу. Всё это без кодирования и деплоя.
