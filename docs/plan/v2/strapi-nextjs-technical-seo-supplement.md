# Техническое SEO-дополнение: Strapi + Next.js архитектура

> Дополнительный документ к архитектуре Visual Page Builder. Фокус: только архитектурно-технические решения, которые нужно заложить заранее для сильного SEO, высокой скорости, корректной индексации и минимизации дорогих переделок.

---

## 1. Цель документа

Техническое SEO должно быть встроено в архитектуру с самого начала, а не добавляться поверх готового сайта. Google прямо указывает, что Core Web Vitals входят в сигналы page experience, а page experience учитывается системами ранжирования [cite:78]. Для хорошего пользовательского опыта целевые пороги остаются такими: LCP до 2.5 секунды, INP до 200 мс и CLS до 0.1 [cite:75][cite:84].

Для headless-архитектуры это означает, что SEO зависит не только от метатегов, но и от способа генерации HTML, стратегии изображений, структуры URL, правил canonical/hreflang, корректной индексации, схем данных и политики кэширования [cite:67][cite:77].

---

## 2. Базовые архитектурные принципы

### 2.1 Рендеринг для SEO

Основа — серверный HTML на первом ответе. Для страниц, важных для органики, лучше использовать SSG или ISR, а не чисто клиентский рендеринг, потому что это улучшает crawlability, скорость первой отрисовки и стабильность индексации [cite:62][cite:71].

Рекомендуемое разделение:

| Тип страницы | Рендеринг | Причина |
|---|---|---|
| Главная, услуги, врачи, акции, статьи | SSG + ISR | Быстрый HTML, хорошая индексация, точечное обновление [cite:22][cite:71] |
| Личный кабинет, защищённые зоны | SSR / dynamic | Не предназначены для SEO |
| Визуальный редактор, admin | Noindex + auth | Не должны индексироваться [cite:77] |

### 2.2 Server Components как дефолт

Для публичного сайта надо использовать React Server Components и App Router как базовый режим, а клиентские компоненты добавлять только там, где реально нужна интерактивность. Это уменьшает JS-бандл, снижает нагрузку на main thread и помогает держать INP в хорошем диапазоне [cite:51][cite:78].

### 2.3 Контент на URL, а не за действиями пользователя

SEO-критичный контент должен присутствовать в HTML страницы по URL, а не открываться только через tabs, accordions, client-side filters или lazy client fetch. Если важный текст, отзывы, цены или FAQ появляются только после JS-действий, это ухудшает доступность для поисковых систем и снижает ценность страницы [cite:77][cite:63].

## 3. Архитектура URL и индексации

### 3.1 Каноническая политика URL

Нужно заранее выбрать и жёстко закрепить одну каноническую схему:

- один основной хост: либо `www`, либо без `www`
- только HTTPS
- единая политика trailing slash
- единый регистр URL, только lowercase slug
- отсутствие дублей через query-параметры для индексируемых страниц

Google указывает, что в canonicalization участвуют HTTPS, redirects и sitemap, поэтому каноническая политика должна быть согласованной везде [cite:86]. Canonical нужно задавать явно на каждой индексируемой странице [cite:67][cite:68].

### 3.2 Структура URL

Рекомендуемый паттерн:

```text
/
/services
/services/[slug]
/doctors
/doctors/[slug]
/promotions
/promotions/[slug]
/articles
/articles/[slug]
```

Правила:

- slug генерируется из названия, но хранится в CMS как отдельное поле
- slug после публикации лучше не менять автоматически
- при изменении slug обязателен 301 redirect со старого URL [cite:77]
- фильтры, сортировки и view-mode не должны порождать индексируемые дубли

### 3.3 Robots policy

`robots.txt` нужен для управления crawl budget, но не для скрытия индексируемых URL. Google отдельно предупреждает: robots.txt не должен использоваться как механизм запрета индексации; для этого нужен `noindex` или авторизация [cite:77].

Базовая политика:

```txt
Disallow: /admin
Disallow: /api
Disallow: /preview
Disallow: /_next/
Allow: /uploads/
Sitemap: https://example.com/sitemap.xml
```

Важно не блокировать CSS, JS и изображения, если они нужны для рендера страницы, потому что Google должен видеть страницу так, как её видит пользователь [cite:83].

## 4. Metadata, canonical, hreflang

### 4.1 Metadata API в Next.js

Для всех публичных страниц метаданные должны генерироваться через `generateMetadata()`, а не вручную через разбросанные `<Head>` вставки. Это делает SEO предсказуемым и централизованным [cite:64][cite:71].

На каждой индексируемой странице должны быть:

- уникальный `title`
- уникальный `description`
- `alternates.canonical`
- Open Graph
- Twitter cards
- robots policy (если страница неиндексируемая)

### 4.2 Canonical как обязательное поле архитектуры

Next.js сам по себе не ставит canonical автоматически, поэтому его надо рассчитывать централизованно в metadata-слое [cite:68]. Для коллекционных сущностей canonical должен строиться на основе одного абсолютного public URL, а не текущего host из запроса.

Пример:

```ts
export async function generateMetadata({ params }) {
  const page = await getDoctor(params.slug);
  const url = `${process.env.NEXT_PUBLIC_SITE_URL}/doctors/${page.slug}`;

  return {
    title: page.seoTitle ?? page.name,
    description: page.seoDescription,
    alternates: {
      canonical: url,
    },
    openGraph: {
      url,
      images: [{ url: page.ogImageUrl }],
    },
  };
}
```

### 4.3 Hreflang

Если мультиязычность вероятна даже не сразу, архитектуру лучше закладывать заранее. Google рекомендует использовать `hreflang` для multilingual-сайтов [cite:77]. У всех локализованных версий должны быть взаимные ссылки друг на друга, и `x-default` должен указывать на дефолтную версию [cite:67].

Если мультиязычность пока не нужна, стоит хотя бы:

- хранить locale в модели страницы
- не зашивать тексты в код
- подготовить URL-схему под `/ru/...`, `/en/...`
- проектировать sitemap с возможностью локализованных записей

## 5. Изображения: производительность и image SEO

### 5.1 Общая стратегия

Изображения — один из самых дорогих классов ошибок. Они напрямую бьют по LCP и CLS, а также влияют на Google Images. Для технического SEO надо сразу закладывать автоматическую обработку медиа: ресайз, современные форматы, контроль качества, размеры, alt, filename и crawlable URLs [cite:82][cite:66].

### 5.2 Что должно происходить при загрузке изображения

На уровне архитектуры у медиа-пайплайна должны быть такие этапы:

1. Загрузка оригинала в Strapi Media Library
2. Генерация нескольких размеров: например `320`, `640`, `960`, `1280`, `1600`
3. Конвертация хотя бы в WebP, желательно и в AVIF там, где это оправдано [cite:62][cite:74]
4. Сохранение исходных размеров `width` и `height`
5. Хранение alt-текста как обязательного или сильно рекомендованного поля
6. Формирование crawlable public URLs, не закрытых robots и не требующих auth [cite:66]

### 5.3 Почему это критично

Google рекомендует, чтобы image URLs были crawlable and indexable, а structured data image properties ссылались на реальные доступные изображения [cite:66]. Для CLS обязательно задавать размеры изображения заранее, чтобы браузер мог зарезервировать место [cite:63][cite:68].

### 5.4 Правила использования изображений на фронте

- Для контентных изображений использовать `next/image` [cite:62][cite:74]
- Для всех изображений задавать `width` и `height` или `fill` с предсказуемым контейнером [cite:63]
- Для LCP-изображения использовать `priority` [cite:71]
- Для изображений ниже первого экрана — lazy loading по умолчанию [cite:65][cite:74]
- Указывать `sizes`, чтобы браузер не тянул слишком большие ресурсы [cite:62]
- У hero-image должен быть отдельно подготовленный desktop/mobile вариант

### 5.5 Форматы и политика качества

Практичный baseline:

- фото: WebP как основной формат, AVIF как дополнительный там, где есть выигрыш [cite:62][cite:72]
- логотипы и иконки: SVG, если это уместно
- прозрачные изображения: WebP/PNG по ситуации
- quality policy на уровне пайплайна, а не руками контентщика

Важно не отдавать оригиналы 4–10 МБ на страницу. Публичный сайт должен использовать derivative-версии, а оригинал хранить отдельно как source asset.

### 5.6 Image SEO-поля в CMS

Для каждого изображения желательно хранить:

| Поле | Зачем |
|---|---|
| alt | Доступность и SEO [cite:79][cite:82] |
| title / caption | Контекст для редактора |
| focal point | Правильный crop для карточек |
| width / height | Борьба с CLS [cite:63] |
| mime / size | Контроль качества |
| original filename | Доп. контекст для медиа-менеджмента |

Если изображение участвует в schema, OG или article/news markup, URL должен быть постоянным и публично доступным [cite:66].

## 6. Core Web Vitals by design

### 6.1 LCP

Чтобы не переделывать потом половину фронта, нужно заранее договориться:

- hero секция не должна зависеть от client-side fetch
- главный текст и hero image должны быть в первом HTML-ответе [cite:62]
- критические шрифты грузить через `next/font`
- не перегружать первый экран carousels, тяжелыми анимациями и видео
- Above-the-fold изображения должны быть заранее оптимизированы и помечены `priority` [cite:71]

### 6.2 CLS

Главные источники CLS — изображения без размеров, late-injected banners, dynamic widgets разной высоты, webfonts и нефиксированные embeds. Хороший baseline — любой виджет должен уметь резервировать место даже до прихода данных [cite:68][cite:75].

Нужно заложить:

- skeleton или fixed placeholder height для async-блоков
- aspect ratio для карточек и медиа
- стабильные размеры кнопок, бейджей, форм
- запрет на вставку контентщиком произвольных iframe без контейнера

### 6.3 INP

INP страдает от тяжёлого клиентского JS. Поэтому:

- как можно больше блоков — server components
- минимум глобальных client providers
- никакой тяжёлой визуальной магии на каждой карточке
- виджеты страницы должны быть независимы и не тащить общий runtime без необходимости

Google сохраняет good threshold для INP на уровне до 200 мс [cite:75][cite:90].

## 7. Sitemap и управление индексацией

### 7.1 Sitemap generation

Sitemap должна генерироваться динамически из реальных канонических URL, а не вручную. В sitemap должны попадать только страницы, которые:

- возвращают `200`
- индексируемы
- являются canonical URL
- не закрыты `noindex`
- реально существуют [cite:67][cite:83]

### 7.2 Что включать

В sitemap включать:

- основные страницы
- страницы услуг
- страницы врачей
- статьи/новости
- акции

Не включать:

- preview URL
- admin
- search results
- параметрические URL фильтров
- страницы с canonical на другой адрес

### 7.3 Lastmod

`lastmod` должен формироваться честно, на основе реальной даты существенного обновления страницы, а не на каждый деплой [cite:67]. Иначе поисковики перестают доверять этим данным.

### 7.4 Отдельные sitemaps

Для масштабируемости лучше сразу проектировать `sitemap index`:

```text
/sitemap.xml
/sitemaps/pages.xml
/sitemaps/doctors.xml
/sitemaps/services.xml
/sitemaps/articles.xml
```

Если later будет multilingual, это проще расширяется.

## 8. Structured data как часть модели данных

### 8.1 Почему это надо закладывать сразу

JSON-LD лучше проектировать одновременно с моделями Strapi, а не как отдельную постобвязку. Google рекомендует использовать supported structured data formats и следить, чтобы данные точно соответствовали контенту страницы [cite:66].

### 8.2 Какие схемы предусмотреть

Минимальный baseline:

| Тип страницы | Schema |
|---|---|
| Весь сайт | `Organization` [cite:69] |
| Хлебные крошки | `BreadcrumbList` |
| Страница врача | `Physician` или `Person` + `MedicalBusiness` контекст |
| Страница услуги | `Service` |
| FAQ-блок | `FAQPage` |
| Статья/новость | `Article` / `NewsArticle` |
| Список отзывов | Осторожно, только если соответствует правилам Google [cite:66] |

### 8.3 Архитектурный принцип

Каждый тип страницы должен иметь свой schema builder:

```ts
buildDoctorSchema(doctor)
buildServiceSchema(service)
buildArticleSchema(article)
```

А не один универсальный "schema helper" на всё, иначе потом начинается хаос. Изображения в schema должны быть релевантны странице и доступны для краулинга [cite:66].

## 9. Внутренняя перелинковка и навигационная архитектура

Google использует ссылки для обнаружения новых страниц и понимания релевантности [cite:79]. Поэтому перелинковка — это техническая задача, а не только редакторская.

Нужно заложить:

- автогенерацию breadcrumb
- блоки «связанные услуги», «другие врачи», «похожие статьи»
- HTML-ссылки, а не кнопки с `onClick`
- нормальные anchor texts, а не «подробнее» везде [cite:79]
- пагинацию или удобную навигацию по большим коллекциям

Если карточка на listing page ведёт на detail page, это должна быть обычная crawlable `<a href>`, а не див с навешанным JS [cite:79].

## 10. Технические требования к виджетам page builder

Чтобы визуальный конструктор не разрушил SEO, у каждого виджета должен быть SEO-контракт.

### 10.1 Что должен уметь каждый виджет

- корректно рендериться на сервере
- не требовать client-side hydration без причины
- не ломать heading hierarchy
- иметь предсказуемую высоту или placeholder
- поддерживать lazy loading для тяжёлых внутренних медиа
- не скрывать основной контент за tabs/accordion по умолчанию, если это ключевой SEO-контент

### 10.2 Widget guardrails

В page builder стоит встроить ограничения:

- только один H1 на страницу
- блок Hero не может повторяться дважды с H1
- noindex-страницы нельзя случайно добавить в sitemap
- контентщик не может загрузить изображение без alt, если оно контентное
- нельзя вставить тяжёлый raw iframe без lazy и контейнера

Это уже не просто UX-ограничения, а техническое SEO governance.

## 11. Контент-модель, которую стоит заложить заранее

Даже если сейчас фокус не на контент-стратегии, в модели страницы должны быть технические поля SEO:

### 11.1 Поля у Page / Doctor / Service / Article

| Поле | Обязательно | Назначение |
|---|---|---|
| slug | Да | Канонический URL |
| seoTitle | Да | Title override |
| seoDescription | Да | Meta description |
| canonicalUrl | Опц. | Для специальных кейсов |
| ogImage | Да | Социальный шаринг [cite:68] |
| noindex | Опц. | Управление индексацией [cite:77] |
| lastReviewedAt / updatedAt | Желательно | `lastmod` и trust |
| locale | На будущее | hreflang |

### 11.2 Global settings

Отдельный Single Type `SEO Settings`:

- site name
- default title template
- default OG image
- robots defaults
- organization schema data
- social profiles
- canonical host

Это снимает хаос, когда такие вещи оказываются разбросаны по коду.

## 12. Кэширование, CDN и media delivery

### 12.1 Кэш-слои

Техническое SEO ломается, если сайт быстрый только локально. Нужно сразу проектировать несколько кэшей:

- CDN/edge cache для статических ассетов
- image cache для derivative-версий
- application cache через ISR
- браузерное кэширование для immutable assets

### 12.2 Политика для медиа

Для изображений нужен длинный cache-control для versioned URLs. Если URL меняется при смене файла, можно смело давать aggressive caching на производные изображения [cite:72].

### 12.3 Компрессия и транспорт

Нужно включить:

- Brotli / gzip для текстовых ресурсов [cite:62]
- HTTP/2 или HTTP/3
- TLS без лишних redirect chains
- минимальное число hops до origin

TTFB сам по себе не является отдельным Core Web Vital, но он напрямую влияет на LCP и общее ощущение скорости [cite:62][cite:75].

## 13. Наблюдаемость и SEO QA

Если SEO-архитектура не измеряется, она ломается незаметно. Поэтому надо заложить:

- Google Search Console
- логирование 404/500
- мониторинг Core Web Vitals в полях, а не только в Lighthouse [cite:63][cite:90]
- регрессионный SEO checklist перед релизом
- автоматическую проверку sitemap, robots, canonical, noindex, schema

### 13.1 Release checklist

Перед выкладкой любого крупного изменения проверять:

- все индексируемые страницы отдают `200`
- canonical присутствует
- в sitemap нет `noindex` или redirect URL [cite:83]
- robots не блокирует критичные ресурсы [cite:77][cite:83]
- изображения имеют размеры и alt
- hero image не ломает LCP
- structured data валидна и соответствует контенту [cite:66]

---

## 14. Что нельзя откладывать на потом

Есть решения, которые надо принять сразу, иначе переделка будет дорогой:

1. **Canonical host policy** — иначе накопятся дубли [cite:86]
2. **Slug lifecycle + 301 strategy** — иначе потеря трафика при редактировании URL [cite:77]
3. **Media pipeline** — иначе тысячи неоптимизированных изображений придётся мигрировать вручную [cite:62][cite:82]
4. **SEO fields в моделях Strapi** — иначе начнётся хаос в metadata
5. **Sitemap/robots архитектура** — иначе индексируются мусорные URL [cite:77]
6. **Widget SEO guardrails** — иначе визуальный конструктор начнёт производить SEO-долг
7. **SSR/SSG-first public architecture** — иначе придётся переписывать рендеринг ради скорости и индексации [cite:62][cite:71]

## 15. Практический baseline для твоей системы

Рекомендуемый baseline для платформы:

- Next.js App Router
- Public pages: SSG + on-demand ISR [cite:22]
- Server Components by default
- `generateMetadata()` для всех публичных роутов [cite:64][cite:71]
- `robots.ts` и `sitemap.ts` с генерацией из Strapi [cite:64]
- единый canonical builder
- image pipeline в Strapi/Sharp: resize + WebP + metadata
- `next/image` для всех контентных изображений [cite:74]
- JSON-LD schema builders по типам страниц [cite:66][cite:69]
- Widget-level SEO constraints в page builder
- отдельный SEO Settings single type в Strapi
- Search Console + field CWV monitoring + release QA

Это даст не просто «сайт, который можно продвигать», а платформу, на которой можно масштабировать контент без системного SEO-долга.
