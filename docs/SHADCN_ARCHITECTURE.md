# Архитектурный проект: Гибридное внедрение Shadcn UI + Radix + Framer Motion

## 1. Цель и Философия (The "Why")
Главная цель интеграции — получить мощные, доступные (a11y) и сложные компоненты для интерактивных сценариев (формы, календари, селекты, модальные окна), сохранив при этом максимальную скорость загрузки и чистоту кода в контентной витрине.

**Ключевой принцип:** Мы используем **Hybrid UI Architecture**.
*   **Dumb Components (Tailwind Native):** Карточки, кнопки навигации, типографика, гриды. 80% приложения. Загружаются мгновенно, не содержат сложного JS.
*   **Smart Components (Radix + Shadcn):** Календари (Datepicker), сложный выбор филиалов (Combobox/Select), модальные формы (Dialog), выпадающие меню профиля (Dropdown). 20% приложения. Базируются на Radix UI.
*   **Orchestration (Framer Motion):** Отвечает *только* за хореографию. Плавная смена роутов, появление модальных окон, раскрытие аккордеонов. Исключено использование Framer Motion для позиционирования блоков (Zero-Layout-Shift).

## 2. Инфраструктура и Зависимости

Для реализации мы развернем следующий стек поверх существующего React + Tailwind:

**Базовые классовые утилиты (уже частично в проекте):**
*   `clsx` + `tailwind-merge` — для умного слияния классов (чтобы `p-4` корректно перебивало `p-2` при переиспользовании).
*   `class-variance-authority` (CVA) — для декларативного описания состояний компонентов (например, кнопок разного размера и цвета).

**Ядро (Headless) и Анимации:**
*   `@radix-ui/react-*` (устанавливается точечно).
*   `framer-motion` (уже установлен, версия 10+).

## 3. Структура директорий (Separation of Concerns)

Чтобы не превратить проект в свалку чужого кода, мы жестко разделим "библиотечные" компоненты и "наши" компоненты.

```text
src/
├── lib/
│   └── utils.ts              # Место обитания функции cn() и форматтеров.
├── components/
│   └── ui/                   # СТРОГО ДЛЯ SHADCN/RADIX. 
│                             # Это "чистая зона". Сюда CLI складывает DatePicker, Dialog, Select. 
│                             # Здесь нет бизнес-логики.
├── shared/
│   └── ui/                   # НАШИ КАСТОМНЫЕ КОМПОНЕНТЫ. 
│                             # Карточки акций (Card), навигация, бейджи. То, что мы пишем руками.
└── widgets/                  # Бизнес-блоки (Header, Footer, CalendarPicker), 
                              # которые оркестрируют UI-компоненты.
```

*Запрещается*: Импортировать что-либо из `src/widgets` внутрь `src/components/ui`. Уровень Shadcn должен оставаться "глупым" дизайн-системой.

## 4. Стратегия Theming (Стилевое слияние)

Shadcn использует концепцию CSS-переменных (`--primary`, `--ring`, `--muted`). 
Мы **НЕ БУДЕМ** слепо копировать дефолтные черно-белые или синие темы Shadcn.

Мы пробросим наши фирменные корпоративные цвета Клиники (изумрудный, фиолетовый) в движок Shadcn:

1.  В `index.css` мы определим глобальные переменные:
    ```css
    :root {
      --primary: 153 52% 41%; /* Наш brand-green (#319773) */
      --primary-foreground: 0 0% 100%;
      --ring: 153 52% 41%; /* Кольцо фокуса при навигации с клавиатуры */
      --radius: 1rem; /* Сохраним сглаженность наших карточек */
    }
    ```
2.  Обновим `tailwind.config.mjs`, чтобы он связал переменные с классами утилит.

## 5. Дорожная карта внедрения (Roadmap)

Внедрение проходит в три этапа (без остановки текущей разработки).

### Фаза 1: Ядро и Утилиты (Текущий Спринт)
1. Установка CLI Shadcn (конфигурация `components.json`).
2. Настройка `cn()` утилиты в `src/lib/utils.ts`.
3. Установка `class-variance-authority`.
4. Интеграция переменных в `index.css` и `tailwind.config`.

### Фаза 2: Точечная замена интерактивных узлов (Priority: High)
Определены **три критические зоны**, где Radix даст мгновенный скачок качества:
1. **Главное Мобильное Меню:** Перевод на `Sheet` (Radix Dialog) с анимацией выезда через Framer Motion (сейчас жесткое открытие).
2. **Селектор Филиала/Города в шапке:** Перевод на `DropdownMenu` (сейчас это самописный блок).
3. **Debug Modal:** Перевод на `Dialog` (улучит доступность).

### Фаза 3: Эпик "Онлайн-запись" (Priority: Highest)
Создание сложного флоу записи пациента:
*   Использование `Calendar` + `Popover` (для выбора даты селекции врача).
*   Использование `Select` (Radix Select с поиском) для выбора специальности.
*   Использование `ScrollArea` для плавного скролла слотов времени.

## 6. Жесткие регламенты (Guardrails)

Для защиты архитектуры от "карго-культа" вводятся правила:

1.  **Strict Selection:** Мы берем компонент из Shadcn **только если** он содержит сложный стейт (aria-состояния, фокус-траппинг, порталы в `<body />`).
2.  **No Boilerplate Abuse:** Мы не устанавливаем `Button` или `Card` из Shadcn, чтобы не засорять код утилитарными компонентами, которые мы и так написали лучше под конкретный дизайн клиники.
3.  **Framer Motion Boundaries:** Используем `AnimatePresence` только на уровне виджетов или страниц. Запрещено вешать `motion.div` в базовых UI-примитивах, чтобы не ломать переиспользование.
