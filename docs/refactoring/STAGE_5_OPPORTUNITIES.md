# Этап 5: Быстрые победы и новые горизонты (A/B & Theming)

## 1. Аудит возможностей
Из-за жесткого хардкода CSS-классов вроде `bg-white`, `bg-gray-50`, `pt-32` в `PageRenderer`, любое изменение корпоративных цветов клиента требует переписывания ядра платформы. Слишком высокое зацепление (coupling).

## 2. Архитектурный План
1. **Theming Variables:**
Заменить "white|gray|dark" на логические токены в конфигурации.
Вместо манипуляций со строкой (`if (bg === 'dark') class += 'bg-gray-900 text-white'`), мы передаём data-атрибуты оболочке блока `data-theme="dark"`, `data-spacing="large"`. Вся логика отображения уйдет на слой CSS/Tailwind (через атрибуты-селекторы).

2. **A/B Tests (Split Testing):**
Иногда клиенту нужно повесить две разных версии виджета и смотреть конверсию. Добавим поддержку вариантов в сам блок:
```typescript
interface PageBlock {
  id: string;
  type: string;
  variants?: {
    id: string;
    weight: number; // probability
    content?: Record<string, any>;
    design?: BlockDesignParams;
  }[];
}
```
Если в `PageBlock` есть `variants`, рендерер рандомно (или на базе куки юзера) выберет нужный вариант и сольет его с основным.

## 3. Шаги реализации:
1. Вынести стили виджета на уровень data-атрибутов оболочки слоя `WidgetWrapper`.
2. Реализовать простой "Split Test Provider" или утилиту в `PageRenderer`, которая может подменять `content` и `design` на базе случайного выбора по весам (`weight`).

## 4. Авто-тесты и валидация
Написание мока A/B теста, верификация применения стиля "dark" через инспектор CSS.
