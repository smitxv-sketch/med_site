# Этап 6: Неочевидные архитектурные возможности (V2.5)

Этот документ описывает "скрытые жемчужины" (hidden gems), которые легко реализовать на базе новой архитектуры, не уходя во взрывную разработку, но дающие колоссальный бизнес-эффект.

## 1. Ссылочные блоки (Global "Reference" Blocks)
**Проблема:** Если один и тот же подвал (Footer), блок "Контакты" или "Осторожно, противопоказания" нужен на 50 посадочных страницах, при изменении телефона в одном месте придется менять 50 документов.
**Решение:** Внедрение специального типа виджета `ReferenceWidget`.
```typescript
{
  id: "ref-1",
  type: "ReferenceWidget",
  config: { refId: "global-footer-block" }
}
```
**Эффект:** `PageRenderer` при встрече такого блока делает lookup в глобальный стор (или контекст) и рекурсивно рендерит нужный блок. Абсолютный DRY (Don't Repeat Yourself) для контент-менеджеров.

## 2. Named Slots (Именованные слоты) вместо просто children
**Проблема:** Массив `children` (из Этапа 2) работает, когда мы просто "складываем" виджеты в ряд внутри контейнера. Но что если нам нужна карточка, куда в `header` нужно положить картинку, в `content` — текст, а в `footer` — кнопку?
**Решение:** Заменить или дополнить `children: PageBlock[]` на `slots: Record<string, PageBlock[]>`.
**Эффект:** Мощнейшая инверсия контроля. Виджет `LayoutCardWidget` просто расставляет слоты, а в CMS (в JSON) мы говорим, что именно отрендерить в конкретном месте карточки.

## 3. Inter-widget Event Bus (Событийная шина)
**Проблема:** Как сделать так, чтобы клик по `PricingWidget` открывал модульное окно, находящееся внутри `ModalWidget`, если у них нет общего состояни (ведь они собираются динамически)? Свалка стейтов на уровне всей страницы — зло.
**Решение:** Внедрить Action-строки.
Например, в кнопке `HeroWidget`: `{ action: "trigger:modal:booking" }`.
`PageRenderer` оборачивает всё в слой `EventBusContext`. Виджет кнопки эмитит событие, а `ModalWidget` слушает глобальное событие `trigger:modal:booking` и открывается.
**Эффект:** Виджеты становятся полностью независимыми (Decoupled), но могут управлять друг другом.

## 4. Rules Engine (Правила персонализации)
**Проблема:** Текущие настройки видимости (`visibilityTarget`, `visibilityAuth`) ограничены. Маркетологу нужно показать виджет акции ТОЛЬКО в выходные дни, или ТОЛЬКО клиентам, пришедшим по UTM-метке `?utm_campaign=black_friday`.
**Решение:** В слое `config` добавить `visibilityRule: string`.
```typescript
config: {
  visibilityRule: "matches(context.utm_campaign, 'black_friday') || user.visitCount > 3"
}
```
**Эффект:** Превращение простого "конструктора лендингов" в полноценный Personalization Engine (как в энтерпрайз-продуктах типа Optimizely или Adobe Target), требующий от разработчика только написания простого безопасного парсера выражений (или использования легковесной библиотеки типа `jexpr`).

## 5. Полный Headless & Edge Computing
**Проблема:** Рендеринг и CMS жестко связаны в рамках одного стора Zustand.
**Решение:** Форкнуть (абстрагировать) `PageRenderer` так, чтобы он не знал ни про какие Zustand-сторы. Он должен быть чистой React-компонентом `(props: { blocks: PageBlock[], context: SiteContext }) => JSX`.
**Эффект:** Это означает, что манифест страницы (JSON) можно будет кэшировать на CDN (Edge). При входе пользователя Next.js / Vite загружает `page.json` из кэша за 10мс, кидает в чистый `PageRenderer`, и страница собирается мгновенно для любого из тысяч генерируемых лендингов.
