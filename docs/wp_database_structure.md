# Структура данных врачей в базе данных WordPress (WP Connector)

В данном документе описана полная структура данных, используемая для хранения и получения информации о врачах из базы данных WordPress (WP). Эта структура используется коннектором WP в сервисе онлайн-записи (`/booking/`).

## 1. Основные таблицы

Информация о врачах хранится в стандартных таблицах WordPress, а также кэшируется в специальной плоской таблице для ускорения работы API.

1. **`wp_users`** — стандартная таблица пользователей WP.
2. **`wp_usermeta`** — стандартная таблица метаданных пользователей WP (ключ-значение).
3. **`wp_doctors_api_cache`** — кастомная таблица, которая генерируется скриптом кэширования (`docs.php`) на основе данных из `wp_users` и `wp_usermeta`.

---

## 2. Связи и идентификация врача

Врач в системе — это пользователь WordPress (запись в `wp_users`), у которого в таблице `wp_usermeta` существует ключ **`id_doctor_qms`** с непустым значением. 

* **Связь таблиц**: 
  * `wp_users.ID` = `wp_usermeta.user_id`
  * `wp_users.ID` = `wp_doctors_api_cache.wp_user_id`

---

## 3. Описание полей в таблицах

### 3.1. Таблица `wp_users`
Используется только для получения базовой информации.
* **`ID`** (int) — Уникальный идентификатор пользователя (врача) в WP.
* **`display_name`** (string) — ФИО врача (отображаемое имя).

### 3.2. Таблица `wp_usermeta` (Метаданные врача)
Здесь хранится вся основная информация. Данные хранятся в формате `user_id`, `meta_key`, `meta_value`.

**Ключи (`meta_key`), относящиеся к врачам:**

* **Идентификаторы:**
  * `id_doctor_qms` — Идентификатор врача в медицинской информационной системе (МИС / QMS). Является главным признаком того, что пользователь — врач.

* **Специальность и должность:**
  * `feed_spec` — Специальность врача (например, "Терапевт"). Если поле пустое, система пытается извлечь специальность из полей `position` или `anonce`.
  * `position` — Должность врача.
  * `anonce` — Краткое описание (анонс) врача.

* **Опыт и стаж:**
  * `feed_start_date` — Дата начала врачебной практики в формате `YYYYMMDD` (например, `20100501`). Используется для автоматического расчета стажа (`experience_years`).
  * `stage` — Текстовое поле стажа (например, "с 2010 года"). Используется как запасной вариант для расчета стажа, если `feed_start_date` не задан (система ищет 4 цифры года).

* **Стоимость и длительность приема:**
  * `feed_price` — Базовая стоимость приема (число).
  * `cost` — Текстовое поле стоимости. Используется, если `feed_price` пуст (система извлекает из него только цифры).
  * `duration` — Длительность приема в минутах.

* **Регалии (Степень, Звание, Категория):**
  * `feed_stepen` — Ученая степень (например, "Кандидат медицинских наук"). Если пусто, система ищет ключевые слова в `position` и `anonce`.
  * `feed_zvanie` — Ученое звание (например, "Профессор", "Доцент"). Если пусто, ищется в `position` и `anonce`.
  * `feed_category` — Врачебная категория (например, "Высшая категория", "Первая категория"). Если пусто, ищется в `position` и `anonce`.

* **Специализация по возрасту:**
  * `feed_childdoctor` — Флаг детского врача (`'1'` = да).
  * `feed_adultdoctor` — Флаг взрослого врача (`'1'` = да).

* **Фотография:**
  Система перебирает следующие ключи в поисках фото: `photo`, `photo_list`, `avatar`, `profile_picture`, `cupp_upload_meta`, `wp_user_avatar`, `user_avatar`, `image`, `picture`, `thumbnail`.
  * Значение может быть прямой ссылкой (`http...`) или ID вложения в WP (тогда URL получается через `wp_get_attachment_url`).

* **Образование (Структурированное):**
  Хранится в виде набора ключей с индексами (`{$idx}`):
  * `feed_edu_{$idx}_feed_edu_dete` — Год (берется первые 4 символа).
  * `feed_edu_{$idx}_feed_edu_org` — Учебное заведение.
  * `feed_edu_{$idx}_feed_edu_spec` — Специальность по диплому.
  * `feed_edu_{$idx}_feed_edu_name` — Тип образования (например, "Базовое", "Ординатура").

* **Текстовые описания:**
  * `education` — Основной текст об образовании.
  * `extraeducation` — Дополнительное образование.
  * `conferences` — Участие в конференциях.
  * `feer_serts` / `feed_serts` / `certificates` / `certs` — Сертификаты.
  * `description` — Полное подробное описание врача.
  * `activities` — Направления деятельности / что лечит врач.
  * `badges` — JSON-строка с бейджами/наградами врача.

### 3.3. Таблица `wp_doctors_api_cache` (Кэш для API)
Эта таблица собирает данные из `wp_users` и `wp_usermeta` в плоскую структуру для быстрой отдачи по API.

* `id` (int) — Primary Key таблицы кэша.
* `wp_user_id` (int) — ID пользователя в WP (`wp_users.ID`).
* `qms_id` (string) — ID врача в МИС.
* `display_name` (string) — ФИО врача.
* `specialty` (string) — Специальность.
* `photo_url` (string) — Ссылка на фото.
* `experience_years` (int) — Вычисленный стаж в годах.
* `price` (int) — Стоимость приема.
* `duration` (int) — Длительность приема.
* `category` (string) — Категория.
* `degree` (string) — Ученая степень.
* `zvanie` (string) — Ученое звание.
* `position` (string) — Должность.
* `is_child_doctor` (int) — `1` или `0`.
* `is_adult_doctor` (int) — `1` или `0`.
* `education_history` (string) — JSON-массив объектов структурированного образования.
* `education_text` (string) — Склеенный HTML-текст из полей `education`, `extraeducation`, `conferences` и сертификатов.
* `description` (string) — Полное описание.
* `anonce` (string) — Краткое описание.
* `activities` (string) — Направления деятельности.
* `badges` (string) — JSON-массив бейджей (включая автоматически сгенерированные бейджи из степени, звания и категории).
* `raw_meta` (string) — JSON-объект со всеми сырыми метаданными пользователя (для отладки и расширения).

## 4. Логика работы коннектора (API)

1. **Сбор данных (PHP-скрипт `docs.php`):**
   Скрипт регулярно (по крону) делает выборку всех пользователей с `id_doctor_qms`, парсит их метаданные (высчитывает стаж, склеивает тексты, определяет фото) и делает `INSERT` / `UPDATE` в таблицу `wp_doctors_api_cache`.

2. **Отдача данных (Node.js backend):**
   Node.js сервер (в `server/services/wpService.ts`) делает GET-запрос к PHP-эндпоинту (`?action=get_doctors`), который просто отдает содержимое таблицы `wp_doctors_api_cache`.
   Затем Node.js парсит JSON-поля (`education_history`, `badges`, `raw_meta`) и отдает готовый массив объектов на фронтенд.
