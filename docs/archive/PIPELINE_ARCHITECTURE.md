# Архитектура и Пайплайн интеграции (Enterprise Integration Pipeline)

Данный документ описывает общую архитектуру системы онлайн-записи (виджета), процесс прохождения данных от разрозненных источников до пользовательского интерфейса, а также регламент внедрения системы в новые клиники.

## 1. Концептуальная архитектура (4 слоя)

Система построена на принципах микросервисной архитектуры и строгой изоляции слоев (Separation of Concerns). Это позволяет легко масштабировать решение на новые клиники с другими МИС и CMS без переписывания пользовательского интерфейса.

### Слой 1: Источники данных (Data Sources)
Сырые, неструктурированные данные из внешних систем клиники.
- **МИС (Медицинская информационная система):** QMS, Инфоклиника, МедАлог и др. Источник расписания, слотов, цен, юридически точных ФИО и привязки к филиалам.
- **CMS (Система управления контентом):** WordPress, Bitrix и др. Источник маркетинговой информации: фотографий, подробных описаний, регалий, истории образования.

### Слой 2: Адаптеры (Anti-Corruption Layer - ACL)
Изолированные модули (коннекторы), задача которых — подключиться к конкретному источнику, извлечь сырые данные и преобразовать их во внутренний, стандартизированный формат системы.
- **WP Adapter:** Парсит базу WordPress (например, сложные сериализованные массивы ACF).
- **MIS Adapter:** Работает с API медицинской системы (например, SOAP/REST запросы к QMS).
- *Документация слоя:* `WP_DATA_MAPPING.md`, `QMS_DATA_MAPPING.md`.

### Слой 3: Гидратация и Разрешение конфликтов (Aggregator / Resolver)
Ядро бэкенда (BFF - Backend for Frontend). Принимает стандартизированные данные от Адаптеров и "склеивает" их в единый объект врача (Каноническую модель).
- **Разрешение конфликтов:** Если ФИО пришло и из МИС, и из CMS, этот слой решает, какому источнику доверять на основе жесткой матрицы приоритетов.
- **Обогащение:** К "сухому" профилю из МИС добавляется "богатая" информация из CMS (фото, стаж, описание).
- *Документация слоя:* `HYDRATION_RULES.md`.

### Слой 4: Пользовательский интерфейс (Frontend / UI Layer)
"Глупый потребитель". React-приложение (виджет), которое ничего не знает о WordPress или QMS. Оно получает идеальный JSON (Каноническую модель) от Слоя 3 и просто красиво его отрисовывает.
- *Документация слоя:* `DOCTOR_UI_DOCS.md`, `WIDGET_USAGE_DOCS.md`.

## 2. Пайплайн внедрения в новую клинику (Onboarding Pipeline)

При подключении новой клиники (с другими системами или структурой базы) процесс внедрения строго регламентирован:

### Шаг 1: Сбор сырых данных (Data Discovery)
1. Выгрузка дампа базы CMS (например, таблицы `wp_users` и `wp_usermeta` для WordPress).
2. Получение примеров ответов от API МИС (JSON/XML со списком врачей и слотами).
3. Сохранение этих данных в эталонные файлы (например, `full.md`, `qms_response.json`).

### Шаг 2: Проектирование Адаптеров (Mapping Design)
1. Анализ сырых данных с помощью ИИ или аналитика.
2. Создание документа маппинга для CMS (например, `[CLINIC]_WP_DATA_MAPPING.md`). В нем фиксируется, в каких полях (или мета-ключах) лежат нужные данные и как их парсить (регулярные выражения, десериализация).
3. Создание документа маппинга для МИС (например, `[CLINIC]_MIS_DATA_MAPPING.md`).

### Шаг 3: Настройка правил слияния (Hydration Config)
1. Согласование с клиникой приоритетов данных (Что важнее: специальность из МИС или с сайта?).
2. Фиксация этих правил в `HYDRATION_RULES.md`.

### Шаг 4: Разработка (Implementation)
1. Написание кода Адаптеров (Слой 2) строго по документам маппинга.
2. Настройка логики слияния (Слой 3) строго по правилам гидратации.
3. *Фронтенд (Слой 4) при этом не модифицируется.*

### Шаг 5: Тестирование и Эксплуатация
1. Проверка работы виджета на тестовом стенде.
2. Встраивание виджета на сайт клиники (см. `WIDGET_USAGE_DOCS.md`).
