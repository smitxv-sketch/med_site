# Дополнительные технические требования (TECH_SPEC_ADDITIONAL)

Этот документ содержит дополнительные идеи, требования и архитектурные решения, которые расширяют основной технический проект (`TECH_SPEC.md`), чтобы не перегружать базовую документацию.

## 1. Максимальное управление контентом (Концепция No-Code для менеджера)

**Цель:** Полностью исключить разработчика из рутинных задач по обновлению контента и настроек сайта. Вся поддержка сайта должна осуществляться силами контент-менеджера через панель управления (Strapi).

**Архитектурные требования:**
*   **Глобальные настройки (Global Settings):** В CMS должна быть создана отдельная Single Type коллекция для управления глобальными параметрами сайта.
*   **Управление микро-контентом:** Все тексты в футере, номера телефонов, email-адреса, адреса клиник, копирайты должны редактироваться из админки.
*   **Динамические ссылки и иконки:** Блок социальных сетей (иконки, ссылки, порядок вывода) должен быть динамическим. Менеджер должен иметь возможность загрузить новую SVG-иконку и указать для нее ссылку без изменения кода фронтенда.
*   **Управление скриптами и аналитикой:** В глобальных настройках должны быть предусмотрены текстовые поля для вставки HTML/JS кодов (счетчики Яндекс.Метрики, Google Analytics, пиксели ВКонтакте, скрипты онлайн-чатов и коллтрекинга). Фронтенд должен безопасно инжектить эти скрипты в `<head>` или `<body>`.

## 3. Управление интерфейсом (Global UI Settings)

Для обеспечения максимальной гибкости и возможности управлять поведением интерфейса без привлечения разработчиков, в архитектуру заложен паттерн **Global Settings**.

### Архитектурное решение:
- На стороне фронтенда создано глобальное хранилище состояния `src/shared/store/uiSettingsStore.ts` (на базе `zustand`).
- Компоненты подписываются на конкретные настройки из этого хранилища и меняют свое поведение реактивно.
- **Интеграция со Strapi:** В будущем при инициализации приложения (или в `App.tsx`) будет происходить запрос к Single Type коллекции `GlobalSettings` в Strapi. Полученные данные будут передаваться в `uiSettingsStore` через метод `set*`.

### Текущие настройки:
1.  **`hideBottomNavOnScroll` (boolean):**
    -   **Описание:** Включает/выключает паттерн "Hide on Scroll Down" для нижнего мобильного меню (`BottomNav`).
    -   **Поведение:** Если `true`, меню плавно уезжает вниз при скролле страницы вниз (освобождая место для контента) и мгновенно появляется при скролле вверх. Если `false`, меню всегда зафиксировано на экране.
    -   **По умолчанию:** `true`.

### Идеи для будущих настроек (Что еще можно вынести в конфиг):
-   **`heroMobileVariant` ('A' | 'B' | 'C'):** Жесткий выбор варианта отображения первого экрана на мобильных устройствах (сейчас переключается кнопкой для A/B теста).
-   **`enableAnimations` (boolean):** Возможность отключить сложные анимации (Framer Motion) для слабых устройств или по соображениям доступности (Accessibility).
-   **`themeColors` (object):** Переопределение базовых корпоративных цветов (Primary, Secondary) прямо из админки (потребует инъекции CSS-переменных в `:root`).
-   **`showPromotionsBanner` (boolean):** Включение/выключение плавающего баннера с акциями поверх всего сайта.
-   **`contactPhones` / `contactEmails`:** Глобальные контактные данные, которые автоматически подтягиваются в шапку, подвал и на страницу контактов.

**Цель:** Автоматическая передача актуальных данных о врачах и клиниках в сервис Яндекс.Доктор для привлечения дополнительного трафика.

**Архитектурные требования:**
*   **Динамическая генерация XML:** Платформа должна уметь генерировать XML-фид в формате, требуемом Яндексом.
*   **Гранулярность фидов:** Фиды должны генерироваться **индивидуально для каждого города и каждого конкретного филиала**.
*   **Связь "Врач - Филиал":** В CMS на странице врача должны быть предусмотрены дополнительные поля для Яндекс.Доктора. Врач должен быть жестко привязан к филиалам, в которых он ведет прием.
*   **Содержимое фида:** В итоговый XML-фид для конкретного филиала должны попадать только те врачи, которые фактически ведут в нем прием, с указанием их специализаций, стажа, стоимости приема и других обязательных параметров.
