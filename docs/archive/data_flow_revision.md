# Ревизия потока данных: от источника до фронтенда

В данном документе описан полный путь данных от первоисточников (МИС и сайт на WordPress) до клиентского интерфейса (виджета), а также проведен анализ полноты и корректности передаваемых данных.

---

## 1. МИС (Медицинская информационная система)

**Что мы запрашиваем и получаем:**
МИС выступает в роли единого источника истины (SSOT) для транзакционных данных и доступности.

*   **Врачи (`getDoctors`):**
    *   `id` (QMS ID, например `ТACdAACATAA`)
    *   `name` (ФИО врача)
    *   `specialty` (Специальность, как она заведена в МИС)
    *   `offerings` (Список услуг/филиалов, где принимает врач):
        *   `id` (ID услуги/филиала)
        *   `branch` (Информация о филиале: ID, название, адрес)
        *   `price` (Актуальная цена приема)
*   **Расписание (`getSlots`):**
    *   `id` (Уникальный ID слота)
    *   `time` (Время приема, например `10:00`)
    *   `date` (Дата приема)
    *   `isAvailable` (Доступность)
    *   `price` (Цена конкретного слота)

**Вывод:** МИС поставляет минимально необходимый, но самый критичный набор данных для записи.

## 2. Сайт на WordPress (WP)

Сайт хранит расширенную "богатую" информацию о врачах (таблицы `wp_users` и `wp_usermeta`). Мы сравнили текущую реализацию в `wpService.ts` с полным дампом полей из `full.md`.

**Что мы реально тянем и как обрабатываем:**
*   **Идентификация:** `ID` (WP ID), `display_name`, `id_doctor_qms` (для связи с МИС).
*   **Специальность (`feed_spec`):** 
    *   *Проблема/Уязвимость:* В БД WP это поле хранится как сериализованный массив ID таксономий (например, `a:1:{i:0;s:4:"1305";}`). Сейчас бэкенд не делает JOIN с таблицей терминов (`wp_terms`), поэтому вместо нормального названия (например, "Педиатр") мы получаем либо этот массив, либо срабатывает фоллбэк на поле `position` (должность).
*   **Фотография (`photo` / `photo_list`):** Хранит ID вложения. Бэкенд делает правильный дополнительный запрос к `wp_posts` для получения реального URL (`guid`).
*   **Стаж (`feed_start_date` / `stage`):** Вычисляется динамически из даты начала работы (формат `YYYYMMDD`) или парсится из текстового поля ("с 1987 года"). Отрабатывает корректно.
*   **Образование:**
    *   *Структурированное:* Собирается из динамических полей ACF `feed_edu_X_...` (год, организация, специальность, тип). Логика парсинга ключей реализована верно.
    *   *Текстовое:* Собирается из полей `education`, `extraeducation`, `conferences` и склеивается с HTML-тегами. Отрабатывает корректно.
*   **Регалии и бейджи:** Степень (`feed_stepen`) и категория (`feed_category`) успешно извлекаются и преобразуются в массив бейджей.
*   **Описание:** Склеивается из `anonce` (или `description`) и `activities` (направления деятельности). Отрабатывает корректно.

**Что есть в `full.md`, но мы НЕ тянем (Пропуски):**
*   `feed_adultdoctor` (Взрослый врач) и `feed_childdoctor` (Детский врач) — **Упущенная возможность!** Эти поля очень полезны для фильтрации на фронтенде или вывода в виде бейджей. Сейчас они игнорируются.
*   `phone` (Телефон) — не нужен для виджета онлайн-записи.
*   `clinics`, `directions`, `services` (Привязки к клиникам и направлениям в WP) — осознанно игнорируем, так как берем 100% актуальные данные из МИС.
*   `fast` (Ссылка на Medflex) — не используется.
*   `feed_price`, `cost` (Цены на сайте) — парсятся в сервисе, но осознанно отбрасываются на этапе гидратации в пользу цен из МИС.

## 3. Слой гидратации (`hydratorService.ts`)

**Как работает:**
Берет массив "сухих" врачей из МИС и массив "богатых" данных из WP. Связывает их по `qms_id` (`misDoc.id === wpDoc.qms_id`).

**Что берет и что отдает на выход:**
*   `id`: остается из МИС (критично для запросов слотов).
*   `databaseId`: сохраняется WP ID (полезно для аналитики).
*   `name`: берется из WP (`display_name`), фоллбэк на МИС.
*   `specialty`: берется из WP, фоллбэк на МИС. *(Здесь проявляется проблема с сериализованным `feed_spec` из шага 2)*.
*   `image`, `experienceYears`, `badges`, `degree`, `category`, `description`, `educationText`, `educationHistory`, `position`: всё берется из WP.

**Вывод:** Слой гидратации работает безупречно с точки зрения архитектуры. Он забирает весь подготовленный контент из WP, оставляя МИС главным по идентификаторам и ценам.

## 4. Фронтенд (Виджет)

Согласно файлу `DOCTOR_UI_DOCS.md`, фронтенду требуются конкретные поля для отрисовки карточек и модальных окон.

**Маппинг потребностей фронтенда и реальных данных:**
*   **Фотография** -> Есть (`image` из WP).
*   **ФИО** -> Есть (`name` из WP/МИС).
*   **Специальность** -> Есть (`specialty` из WP/МИС).
*   **Стаж** -> Есть (`experienceYears` из WP).
*   **Цена** -> Есть (берется на фронте из `offerings[0].price` от МИС).
*   **Степень / Категория / Бейджи** -> Есть (`degree`, `category`, `badges` из WP).
*   **Адрес приема** -> Есть (берется на фронте из `offerings[].branch.address` от МИС).
*   **Описание (О враче)** -> Есть (`description` из WP).
*   **История образования** -> Есть (`educationHistory` / `educationText` из WP).
*   **Расписание (Слоты)** -> Есть (отдельный асинхронный запрос `getSlots` к МИС).

**Итог по фронтенду:** Структура данных, выдаваемая бэкендом после гидратации, на 100% покрывает потребности UI, описанные в документации.
