# Правила гидратации и разрешения конфликтов данных (Hydration Rules)

Этот документ описывает логику слияния (Aggregator / Resolver) данных о врачах из двух независимых источников: Медицинской информационной системы (МИС, например QMS) и Системы управления контентом (CMS, например WordPress).

## 1. Каноническая модель врача (Canonical Doctor Model)
Это идеальный JSON, который ожидает получить Frontend (виджет). Он формируется на бэкенде (BFF) после слияния данных из МИС и CMS.

```json
{
  "id": "12345", // Строгий ID из МИС
  "name": "Иванов Иван Иванович", // ФИО из МИС
  "specialty": "Врач-невролог", // Специальность из CMS (или МИС)
  "photoUrl": "https://...", // Фото из CMS
  "experienceYears": 15, // Стаж из CMS
  "price": 5000, // Цена из МИС
  "position": "Врач высшей категории", // Должность из CMS
  "isChildDoctor": true, // Флаг из CMS
  "isAdultDoctor": true, // Флаг из CMS
  "education": [...], // Массив образования из CMS
  "slots": [...] // Массив слотов из МИС
}
```

## 2. Матрица приоритетов (Priority Matrix)

При слиянии данных из двух источников неизбежны конфликты (дублирование). Ниже описаны жесткие правила, какому источнику доверять для каждого поля канонической модели.

| Поле Канонической Модели | Приоритетный источник | Резервный источник (Fallback) | Обоснование (Почему так?) |
| :--- | :--- | :--- | :--- |
| **`id`** | **МИС** | — | ID врача в МИС — это первичный ключ для записи на прием. ID из CMS (например, ID пользователя WP) используется только для маппинга, но на фронт не отдается. |
| **`name`** (ФИО) | **МИС** | CMS | В МИС хранятся юридически точные ФИО (как в паспорте), необходимые для формирования чеков и договоров. В CMS могут быть опечатки или сокращения. |
| **`specialty`** | **CMS** | МИС | В МИС специальность часто техническая (например, "Врач-терапевт участковый"). В CMS маркетологи пишут понятнее для пациента (например, "Терапевт, кардиолог"). |
| **`photoUrl`** | **CMS** | — | МИС редко хранит качественные фотографии. CMS — основной источник медиа-контента. |
| **`experienceYears`** | **CMS** | — | В МИС стаж часто не ведется или ведется криво. В CMS это важное маркетинговое поле. |
| **`price`** | **МИС** | CMS | Цена в МИС динамическая (зависит от услуги, времени, скидок). Цена в CMS часто устаревает. МИС — единственный источник финансовой правды. |
| **`position`** | **CMS** | — | Должность, регалии (КМН, профессор) — это маркетинговая информация, которая красиво оформляется в CMS. |
| **`isChild/Adult`** | **CMS** | — | Флаги приема детей/взрослых удобнее настраивать контент-менеджеру в CMS. |
| **`education`**, **`certs`** | **CMS** | — | Подробная история обучения и сертификаты хранятся только на сайте. |
| **`slots`** | **МИС** | — | Расписание — это суть МИС. CMS ничего не знает о реальном графике врача. |

## 3. Алгоритм слияния (Hydration Process)

1. **Запрос в МИС:** Бэкенд запрашивает список врачей (или конкретного врача) и их слоты из МИС. Получаем базовый массив объектов (ID, ФИО, Цена, Слоты).
2. **Маппинг (Связывание):** Бэкенд ищет соответствие между врачом из МИС и врачом в CMS. 
   - *Ключ связывания:* Обычно это СНИЛС, ИНН, или специальное поле `mis_id` в профиле врача в CMS. Если такого поля нет, связывание происходит по точному совпадению ФИО (менее надежно).
3. **Обогащение (Enrichment):** Если связь найдена, бэкенд берет "сухой" объект из МИС и обогащает его "богатыми" данными из CMS (фото, стаж, описание) согласно Матрице приоритетов.
4. **Разрешение конфликтов:** Если поле есть и там, и там (например, `specialty`), бэкенд перезаписывает значение из МИС значением из CMS (согласно Матрице).
5. **Отдача на Фронт:** Сформированная Каноническая модель отправляется в виджет.
