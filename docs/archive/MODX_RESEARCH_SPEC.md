# ТЗ на исследование и разработку универсального API для миграции/синхронизации (MODX)

## 1. Контекст задачи
Мы разрабатываем новую медицинскую платформу на базе Strapi и React. Нам необходимо реализовать процедуру **перманентной синхронизации** данных с текущего работающего сайта на MODX (Санкт-Петербург). 

Задача для ИИ-агента: провести глубокое исследование базы данных MODX, выявить структуру хранения ключевых сущностей и предложить архитектуру "всемогущего" REST API, который позволит бесшовно забирать данные в нашу новую PIM-систему (Strapi).

## 2. Этап 1: Исследование (Discovery)
Агенту предоставляется доступ к БД MODX. Необходимо выполнить следующие шаги:

### 2.1. Анализ сущностей
Изучить таблицы (стандартные `modx_site_content` и кастомные, если есть) и TV-параметры (`modx_site_tmplvar_contentvalues`), чтобы найти:
*   **Врачи:** Где лежат ФИО, фото, специализации, стаж. Как реализована привязка к филиалам.
*   **Услуги:** Где лежат тексты описаний, названия. 
*   **Уникальный ключ:** Найти поле, в котором хранится **артикул (SKU)** услуги (это критически важно для синхронизации цен из МИС).
*   **Категории/Направления:** Как услуги распределены по папкам или тегам (Гастроэнтерология, Гинекология и т.д.).

### 2.2. Анализ связей
Выявить, как в текущей БД реализованы связи:
*   **Врач <-> Услуги:** Как связаны конкретные специалисты с услугами, которые они оказывают.
*   **Услуги <-> Места размещения:** В каких разделах сайта выводится одна и та же услуга.

## 3. Этап 2: Проектирование "Всемогущего" API
На основе исследования агент должен составить ТЗ на разработку REST API на стороне MODX. 

### Требования к API:
*   **Универсальность:** API не должен быть жестко ограничен текущей структурой Strapi. Он должен уметь отдавать "сырые", но структурированные данные по любому запросу.
*   **Метод получения:** Поддержка фильтрации (например, получить всех врачей направления X) и пагинации.
*   **Идентификация:** Все объекты должны отдаваться с их внутренними ID MODX и внешними SKU (для услуг).
*   **Инкрементальность:** Возможность запрашивать только измененные данные (например, через поле `editedon`).

## 4. Ожидаемый результат от Агента
1.  **Отчет об архитектуре БД MODX:** Карта таблиц и полей, отвечающих за врачей, услуги и связи.
2.  **Предложение по Pipeline синхронизации:** Как именно (с какой частотой, по каким ключам) данные должны перетекать в Strapi.
3.  **Спецификация эндпоинтов API:** Список URL и формат JSON-ответов, которые должен реализовать MODX для обеспечения полной синхронизации.
