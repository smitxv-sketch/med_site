# Архитектура Frontend-приложения (Enterprise Level)

Для обеспечения масштабируемости, поддержки и удобной интеграции с Headless CMS (Strapi) в будущем, мы выбираем архитектурную методологию, основанную на **Feature-Sliced Design (FSD)** или модульном Domain-Driven Design (DDD). Это стандарт де-факто для крупных React-приложений.

## 1. Технологический стек (Enterprise Standard)

*   **Core:** React 18 + TypeScript (Строгая типизация обязательна).
*   **Сборщик:** Vite (Максимальная скорость разработки).
*   **Роутинг:** React Router v6 (Поддержка вложенных маршрутов и лоадеров).
*   **Server State (Работа с API):** TanStack Query (React Query). Идеально для кэширования, инвалидации данных и работы со Strapi в будущем.
*   **Client State (Локальное состояние):** Zustand (Легковесный, быстрый, без бойлерплейта Redux).
*   **Стилизация:** Tailwind CSS (Utility-first подход, легко масштабируется).
*   **UI-Kit (Базовые компоненты):** Компоненты на базе Radix UI / shadcn ui (Доступность из коробки, полный контроль над стилями).
*   **Анимации:** Framer Motion (Для плавных "нативных" переходов и открытия меню).

## 2. Структура директорий (Feature-Sliced Design)

Проект разделен на слои (Layers) с однонаправленной зависимостью (нижние слои не могут импортировать верхние).

```text
src/
├── app/                    # Инициализация приложения (Слой 1)
│   ├── providers/          # React Query, Router, Theme providers
│   ├── styles/             # Глобальные стили (Tailwind imports)
│   └── App.tsx             # Корневой компонент
│
├── pages/                  # Страницы приложения (Слой 2)
│   ├── home/               # Главная страница
│   ├── doctors/            # Каталог врачей
│   └── services/           # Каталог услуг
│
├── widgets/                # Самостоятельные блоки (Слой 3)
│   ├── header/             # Шапка сайта
│   ├── mobile-menu/        # Мобильное меню (Бургер)
│   └── doctor-card/        # Карточка врача (собирает сущность и фичи)
│
├── features/               # Пользовательские сценарии (Слой 4)
│   ├── auth/               # Авторизация
│   ├── book-appointment/   # Запись на прием
│   └── search-doctors/     # Поиск врачей
│
├── entities/               # Бизнес-сущности (Слой 5)
│   ├── doctor/             # Сущность "Врач" (модели, UI-компоненты, API-хуки)
│   ├── service/            # Сущность "Услуга"
│   └── clinic/             # Сущность "Клиника"
│
└── shared/                 # Переиспользуемый код (Слой 6)
    ├── api/                # Базовый клиент (Axios/Fetch)
    ├── config/             # Переменные окружения
    ├── lib/                # Утилиты, хелперы
    └── ui/                 # UI-Kit (Button, Input, Modal, Typography)
```

## 3. Принципы Enterprise-разработки в нашем проекте

1.  **Изоляция (Low Coupling):** Изменение логики в карточке врача (`entities/doctor`) не должно ломать карточку услуги (`entities/service`).
2.  **Высокое зацепление (High Cohesion):** Все, что связано с врачом (типы, API-запросы, UI-компоненты отображения), лежит в одной папке `entities/doctor`.
3.  **API-Agnostic UI:** UI-компоненты ничего не знают о том, откуда берутся данные. Они принимают пропсы. Это позволит нам легко переключиться с моковых JSON-файлов на реальный REST/GraphQL API от Strapi.
4.  **Mocking Strategy:** На этапе прототипа все данные будут лежать в `shared/api/mocks`. Мы напишем сервисы, которые будут имитировать задержку сети (Network Latency), чтобы сразу видеть состояния загрузки (Скелетоны/Спиннеры) и проектировать UX с учетом реальных условий.

## 4. План реализации (Фаза 1: App Shell)

1.  Настройка базовых `shared/ui` компонентов (Кнопки, Иконки, Контейнеры).
2.  Создание `widgets/header` (Плавающая шапка).
3.  Создание `widgets/mobile-menu` (Моментальное меню с аккордеонами).
4.  Создание `widgets/bottom-nav` (Нижняя панель навигации).
5.  Сборка `pages/home` (Визуальный хаб услуг).
