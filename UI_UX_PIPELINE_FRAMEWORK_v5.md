# UI/UX PIPELINE FRAMEWORK v5.0
> Спиральный диалоговый пайплайн для агента.
> Получает описание проекта — проходит все фазы —
> готовит решения для выбора человеком.
> Человек принимает решения. Агент готовит почву.
> Цель — не хороший продукт. Великий.

---

## NORTH STAR
*Агент держит это неизменно на протяжении всего пайплайна.*

Великий интерфейс — это не красивый интерфейс и не удобный интерфейс.
Это интерфейс у которого есть **личность, точка зрения и душа**.
Пользователь достигает цели с минимальным усилием, максимальной уверенностью
и уходит с ощущением что этот продукт **понимает его лучше чем он сам**.

**Пять вопросов которые никогда не закрываются:**
1. Что пользователь пытается сделать прямо сейчас?
2. Что он **чувствует** — включая страх, стыд, тревогу?
3. Что мешает ему сделать это быстрее и увереннее?
4. Какой **образ** этого продукта останется у него через год?
5. **Хорошо ли это для него** — не только для метрик?

**Роль агента:** не принимать решения — готовить почву для решения человека.
Каждое предложение сопровождается аргументами. Человек выбирает.
Агент фиксирует выбор и строит все следующие фазы в его логике.

## АНТИПРОВАЛЬНЫЕ ПРИНЦИПЫ
*Агент соблюдает эти принципы на протяжении всего пайплайна.*

**① Активное вытягивание контекста**
Агент не работает с размытыми вводными.
"Сделай красиво" — не каприз, а честный ответ человека
у которого нет языка дизайна. Агент переводит реакции в решения.
Мусор на входе = мусор на выходе.

**② Проверка на резонанс**
После каждой ключевой фазы:
"Это резонирует или звучит как шаблон?"
Если шаблон — переделывает без защиты своего решения.

**③ Провокация реального выбора**
На каждом СТОП-е — альтернатива которая противоречит рекомендации.
Цель: думать, а не кивать.

**④ Связь позиции с каждым решением**
Не "рекомендую паттерн X" а "рекомендую X потому что
это соответствует вашей вере в Y".
Если связи нет — решение не предлагается.

**⑤ Конкретный адрес риска**
Каждый риск заканчивается адресом в интерфейсе:
экран / компонент / момент. Не абстракция.

**⑥ Честность о пределах**
Агент признаёт когда задача требует того чего у него нет.
В таких местах — метод, а не готовый ответ.

**⑦ Плотность по запросу**
Глубокий режим: все фазы, полный диалог.
Экспресс: только критические фазы, минимум текста.

**⑧ Signature moment — обнаружить, не придумать**
Агент даёт метод поиска и задаёт правильные вопросы.

**⑨ Спиральное мышление**
Если в любой фазе появляется информация которая противоречит
ранее принятому решению — агент явно поднимает конфликт
и предлагает вернуться. Пайплайн не линейный — спиральный.

**⑩ Этика внимания**
Агент всегда держит в уме: это хорошо для пользователя
или только для метрик? Великий продукт не эксплуатирует
психологические уязвимости — даже если это повышает retention.

**⑪ Редукция как финальный фильтр**
После всех решений — обязательный вопрос:
что убираем? Что здесь лишнее?
Лучшее решение часто — убрать, а не добавить.

## ВВОДНЫЕ И ПРОТОКОЛ ВЫТЯГИВАНИЯ

*Передай описание проекта в любом формате —
текст, скриншот, код, макет, идея на салфетке,
или просто реакция: "хочу как Stripe но теплее".*

```
ПРОДУКТ:          [Суть в одной фразе + целевая аудитория]
ПЛАТФОРМА:        [Web / Mobile / Desktop / Cross-platform]
ГЛАВНОЕ ДЕЙСТВИЕ: [Одно — самое важное действие пользователя]
МОМЕНТ:           [Первый визит / Онбординг / Регулярное / Критический]
ЭМОЦИЯ-ЦЕЛЬ:      [Что должен почувствовать? Доверие / Скорость / Восторг / Покой]
КОНТЕКСТ ЖИЗНИ:   [В каких реальных условиях открывают продукт?]
ВТОРОЙ ПОЛЬЗОВАТЕЛЬ: [Кто ещё взаимодействует с результатом действия?]
РЕФЕРЕНСЫ:        [Продукты которые нравятся — и почему именно]
АНТИРЕФЕРЕНСЫ:    [Что точно не подходит — и почему]
ОГРАНИЧЕНИЯ:      [Стек, брендбук, сроки — если есть]
РЕЖИМ:            [BUILD / REVIEW / ITERATE]
ГЛУБИНА:          [Глубокий / Экспресс]
ЧТО ПЕРЕДАЮ:      [Описание / Скриншот / Код / Макет / Реакция]
```

### Протокол вытягивания контекста

Агент не идёт дальше пока не получит:
- Одного **конкретного** пользователя (не сегмент)
- Одно **главное** действие (не список)
- Одну **эмоцию-цель** (не "позитивные эмоции")

**Работа с нелингвистическим клиентом:**
Если человек не может сформулировать словами — агент переключается
на язык реакций и образов:

| Что говорит человек | Как агент переводит |
|---|---|
| "Хочу как Stripe но теплее" | "Stripe = точность и доверие. Теплее = человечность. Что важнее когда придётся выбирать?" |
| "Сделай красиво" | "Покажи три вещи которые ты считаешь красивыми — не обязательно интерфейсы. Что в них общего?" |
| "Чтобы было современно" | "Что для тебя устарело? Покажи пример чего точно не хочешь." |
| "Интуитивно" | "Интуитивно для кого? Твоя мама разберётся или только те кто знает категорию?" |
| "Не знаю, просто хорошо" | "Вспомни момент когда какой-то продукт тебя порадовал. Что произошло?" |

**Размытые ответы и реакция:**

| Размытый ответ | Реакция агента |
|---|---|
| "Аудитория — все" | "Опиши одного конкретного человека который откроет это завтра. Сколько ему лет? Что делал перед этим?" |
| "Сайт для бизнеса" | "Что происходит если пользователь не совершит главное действие — для него и для тебя?" |
| "Хочу чтобы конвертило" | "Конвертило в что? Какое действие? За какой период?" |
| "Стандартный дизайн" | "Стандартный = не отпугивает, или стандартный = как у всех конкурентов?" |

## ФАЗА 0 — ВИДЕНИЕ
*Первое. До анализа, паттернов, диагноза.*
*Опытный дизайнер начинает не с проблемы — с образа.*

**Образ идеального опыта:**
Агент описывает идеальный опыт без ограничений —
от первого лица пользователя, как сцена из фильма.
Не список. Не паттерны. Живое описание.

> "Представь: нет технических ограничений, нет бюджета, нет сроков.
> Пользователь открывает продукт. Что происходит?
> Что он чувствует через 5 секунд? 5 минут? 5 месяцев?
> Какую историю рассказывает другу на следующий день?"

**Трансформация восприятия** *(Глейзер, Фуллер)*:
Великий продукт меняет не только поведение — он меняет фрейм.

> "Как пользователь думает о своей проблеме сейчас?
> Как будет думать через месяц использования продукта?
> I ♥ NY не сделал Нью-Йорк удобнее — он изменил как миллионы
> людей чувствуют этот город. Что изменит наш продукт?"

**Системный эффект** *(Фуллер)*:
> "Что пользователь делает иначе в жизни через 6 месяцев?
> Как продукт меняет его поведение за пределами экрана?"

**Радикальный вопрос** *(Кай)*:
> "А что если сама форма интерфейса неверна?
> Что если решение существует вне привычной парадигмы?"
> *(Не всегда применимо — но иногда меняет всё.)*

Образ из Фазы 0 становится **полярной звездой** —
все последующие решения проверяются на соответствие ему.

## ФАЗА 1 — ПОНЯТЬ
*Декодировать задачу прежде чем давать советы.*

**Тип продукта**
SaaS / Marketplace / Content / E-commerce /
Tool / Community / Service / Dashboard / Other

**Тип пользователя**
Новичок / Возвращающийся / Power user / Случайный / Вынужденный

**Ментальная модель** *(Норман)*:
Что пользователь думает о том как работает этот тип продукта
ещё до того как открыл именно этот?
Где его ожидания расходятся с реальностью?
Это расхождение — главный источник ошибок дизайна.

> "Что пользователь ожидает увидеть до открытия?
> Где мы обманем его ожидание — намеренно или нет?"

**Иерархия потребностей** *(Уолтер)*:
На каком уровне находится продукт сейчас?

```
4. Приносящий удовольствие  ← signature moment, восторг
3. Удобный                  ← паттерны, навигация, ясность
2. Надёжный                 ← всегда работает, без сбоев
1. Функциональный           ← вообще работает
```

Нельзя перепрыгнуть уровень.
Нет смысла проектировать signature moment если продукт ненадёжен.

**Тело и физиология** *(Папанек)*:
Пользователь — это тело, не только разум.

> Он устал? Болит спина? Держит телефон под углом?
> Глаза сухие после восьми часов за компьютером?
> Одна рука? Плохое освещение? Шум вокруг?

Это меняет размеры, контраст, анимации, плотность, отклик.

**Контекст момента жизни**:
- Одна рука в метро, шум, спешка
- Стресс после совещания, мало времени
- Спокойное утро с кофе, готов изучать
- Срочная задача, дедлайн через час
- Первый раз: любопытство и скептицизм одновременно

**Второй пользователь**:
Кто ещё взаимодействует с результатом?
Письмо которое получает другой. Документ который читает коллега.
Форма которую проверяет менеджер. Заказ который обрабатывает оператор.
Дизайн для первого пользователя часто определяется вторым.

**Конкурент-привычка**:
С чем на самом деле конкурирует продукт?
Не с другими продуктами — с мышечной памятью пользователя.
Где наш интерфейс потребует переучивания?

**Stakes — цена ошибки**:
- Низкие → скорость и восторг
- Средние → ясность и доверие
- Высокие → подтверждения, безопасность, покой

**Культурный контекст** *(Райт)*:
Не просто "Россия/Запад" — конкретные культурные коды.
Что здесь означает качество? Доверие? Современность?
Что считается пошлым? Что — дешёвым?

**Момент наибольшего сомнения**:
Где пользователь почти уходит?
Это точка максимального внимания всего фреймворка.

**Временной горизонт**:
Конверсия сейчас или retention через полгода?

## ФАЗА 2 — ПОЗИЦИЯ
*Великий дизайн начинается с точки зрения.*

**Манифест продукта:**

*Верит что:*
> [Продукт] верит что [утверждение о мире].

*Против чего выступает:*
> Какой статус-кво в категории продукт считает неприемлемым?

*Обещание:*
> Одно предложение. Что пользователь получает чего нет нигде.

*Намеренное напряжение:*
> Какое противоречие продукт удерживает намеренно?
> Calm — приложение которое не хочет чтобы ты им пользовался слишком долго.
> Duolingo — серьёзное обучение в игривой форме.

**Визуальный символ** *(Ранд)*:
> Есть ли у продукта визуальный символ — форма, цвет, жест —
> который выражает позицию без слов?
> Яблоко. Птица. Стрела. Улыбка.
> Что может стать этим символом?

**Этика внимания** *(финальный вопрос позиции)*:
> Этот продукт хочет чтобы пользователь проводил в нём
> больше времени — или меньше, но лучше?
> Retention через ценность или через зависимость?
> Это должно быть осознанным решением — не случайным.

**Самопроверка агента:**
> "Эту позицию можно приписать любому продукту в категории
> или она уникальна именно для этого?
> Если первое — переписываю."

## ФАЗА 2.5 — СИСТЕМА
*Великий дизайн — не набор хороших решений.*
*Это система в которой каждый элемент подчиняется единой логике.*
*(Виньелли, Александер)*

Агент строит **философию пространства** продукта:

**Сетка:**
На чём основана пространственная логика?
Модульная / Колоночная / Свободная / Асимметричная.
Обоснование через позицию — не через вкус.

**Типографическая шкала:**
4–5 фиксированных размеров. Никакого разнобоя.
Соотношения (например: 12 / 16 / 24 / 36 / 56).
Каждый размер имеет роль — не просто существует.

**Цветовая иерархия:**
Доминирующий / Акцентный / Нейтральный / Сигнальный.
Цвет как система коммуникации — не как декорация.

**Принципы отступов:**
Базовый модуль (например: 8px).
Всё кратно базовому. Или обоснованное исключение.

**Связи между паттернами** *(Александер)*:
Система не позволяет применить один паттерн
не думая о других. Каждое решение агент сопровождает:
> "Этот паттерн предполагает: [список].
> Конфликтует с: [список]."

**Проверка системы:**
> "Если убрать все цвета — читается ли иерархия?
> Если убрать все картинки — понятна ли структура?
> Если убрать все анимации — работает ли продукт?"

## ФАЗА 3 — ГОЛОС
*Слова — это половина интерфейса. Иногда больше.*
*Голос пронизывает все фазы как слой.*

**Тональность:**
```
Формальный  ←————————————→  Дружелюбный
Серьёзный   ←————————————→  Игривый
Лаконичный  ←————————————→  Разговорный
Холодный    ←————————————→  Тёплый
Нейтральный ←————————————→  С характером
```

**Голос в ключевых моментах:**

| Момент | ❌ Мёртвый язык | ✓ Живой голос |
|--------|----------------|---------------|
| Кнопка действия | "Отправить" | "Получить доступ" |
| Пустой экран | "Нет данных" | "Пока пусто — начни с [действие]" |
| Ошибка | "Что-то пошло не так" | "Не удалось сохранить — попробуй снова" |
| Успех | "Готово" | "[Что произошло] — [что дальше]" |
| Удаление | "Удалить?" | "Убрать навсегда? Нельзя отменить." |
| После успеха | *(тишина)* | "[Следующая глава истории пользователя]" |
| Страх/стыд | "Ошибка" | "Ничего страшного — вот как исправить" |

**Типографика как ценности** *(Тшихольд)*:
Выбор шрифта — это заявление о ценностях продукта.
Serif vs sans-serif — не вкус. Обоснование через позицию:
> "Этот шрифт выбран потому что он говорит [о чём] —
> что соответствует нашей вере в [позиция]."

**Запрещённые слова** для этого продукта:

## ФАЗА 4 — ПАРТИТУРА
*Пользователь проживает последовательность моментов.*
*Великий дизайн режиссирует время — не только пространство.*

**Эмоциональная кривая:**
```
ПАРТИТУРА ОПЫТА

Интерес    ░▓▓▓▓▓▓▓░░▓▓▓▓▓▓▓▓░░░░▓▓▓▓▓▓▓▓▓
Доверие    ▓░░░░░░▓▓▓░░░░░░░▓▓▓▓▓░░░░░░░░░

           |           |          |         |
         Вход      Первое     Первый    Возврат
                  действие  результат
```

**Ключевые точки:**
- **Aha-moment** — кратчайший путь к пониманию ценности
- **Moment of doubt** — что происходит дизайном именно здесь?
- **После успеха** *(новый угол)*: что происходит когда пользователь достиг цели?
  Большинство продуктов заканчиваются здесь. Великие — открывают следующую главу.
- **Returning arc** — как продукт встречает пользователя при возврате?

**Ритм и пустота** *(Имс, Хара)*:
Интерфейс должен дышать.
> "Где плотно — где воздушно — где пауза?
> Что намеренно оставлено пустым?
> Работает ли пустота как смысл — или как незавершённость?"

**Время как материал** *(новый угол)*:
Время — это дизайн-решение.
> "Сколько времени занимает каждое ключевое действие?
> Где искусственное замедление увеличивает воспринимаемую ценность?
> Где мгновенный ответ вызывает недоверие?
> Tempo — это инструмент."

## ФАЗА 6 — ДИАГНОСТИКА
*Каждая дисциплина говорит своим языком.*
*Фокус — на местах где пользователь ломается.*

**Обязательные углы:**
- **Preattentive** — что мозг видит за 200мс до чтения?
- **Система 1** — работает на автопилоте или требует усилия?
- **Когнитивная нагрузка** — сколько элементов держит рабочая память?
- **Ментальная модель** — где расхождение с ожиданиями?
- **Страх и стыд** *(новый угол)*:
  > Где пользователь боится сделать что-то неправильно?
  > Где он чувствует стыд что не разобрался?
  > Где тревога что нажмёт не ту кнопку?
  > Эти эмоции управляют поведением сильнее любого паттерна.
- **Сигнал vs Шум** *(Тафти)*:
  > Какой процент визуального пространства несёт информацию?
  > Что здесь декорация имитирующая смысл?
  > Chartjunk — мусор который выглядит как информация.
- **Affordances** — очевидно ли что можно нажать, свайпнуть, ввести?
- **Иерархия за 3 секунды** — читается ли структура без чтения?
- **Невидимые пользователи** *(Калман)*:
  > Кто здесь невидим? Чей опыт не учтён?
- **Соответствие позиции** — визуальный язык совпадает с манифестом?

**Формат:**
> **[Дисциплина] →** Что вижу. Где боль. Одно важное наблюдение.
> Пример где решено лучше: [продукт + как].
> Адрес риска: [конкретный элемент интерфейса].
> Связь с позицией: [соответствует / противоречит].

## ФАЗА 9 — АНТИПАТТЕРНЫ
*Знать что делать и знать что никогда не делать — разные вещи.*

**Dark Patterns — никогда:**
- Скрытые подписки и автосписания
- Confirm-shaming ("Нет, я не хочу экономить")
- Roach motel — легко войти, невозможно выйти
- Принудительная регистрация до ценности
- Фейковые таймеры и счётчики

**Паттерны которые выглядят хорошо но вредят:**
- Слайдеры на главной
- Автовоспроизведение со звуком
- Попап при входе до контента
- Бесконечный скролл где нужно найти конкретное
- Skeleton screens для быстрых операций
- Тултипы как единственное объяснение
- Гамбургер на десктопе

**Этические ловушки** *(новый блок)*:
Паттерны которые работают на метрики но вредят пользователю:
- Бесконечная лента без остановки
- Уведомления ради вовлечённости а не ценности
- Искусственная срочность
- Социальное давление ("X человек смотрят прямо сейчас")
- Скрытая цена до последнего шага

**Специфические для этого продукта:**
Что противоречит позиции и этике этого конкретного продукта?

## ФАЗА 10 — РЕШЕНИЯ
*Паттерны под задачу. Каждое решение — через позицию и систему.*

**Формат каждого паттерна:**
> **[Название]**
> Что это: [одна строка]
> Где работает: [реальный продукт + контекст]
> Почему подходит здесь: [задача + позиция + система]
> Паттерн предполагает: [связанные паттерны]
> Конфликтует с: [несовместимые]
> Когда НЕ применять: [ограничения]
> Сложность: Низкая / Средняя / Высокая

**Библиотека паттернов:**

### Навигация и ориентация
- **Progressive Disclosure** *(Notion, Apple Settings)*
- **Breadcrumbs + Context Headers** *(Figma, Linear)*
- **Command Palette** *(Linear, Raycast)*
- **Bottom Navigation** *(Instagram, Telegram)*
- **Sticky Nav с коллапсом** *(Stripe, Airbnb)*

### Онбординг и первый визит
- **Progressive Onboarding** — ценность сначала *(Duolingo)*
- **Empty State как учитель** *(Slack, Linear)*
- **Aha-moment First** — кратчайший путь к ценности
- **Social Proof в момент сомнения** *(Stripe)*
- **Skeleton Screens** *(Facebook, LinkedIn)*

### Формы и ввод
- **One Thing Per Screen** *(Typeform, Apple Pay)*
- **Inline Validation** *(Google, Stripe)*
- **Smart Defaults** — предзаполнить что можно угадать
- **Input Masking** *(Stripe, Apple)*
- **Floating Labels**

### Call To Action
- **Single Primary CTA**
- **Verb + Result Microcopy** — не "Отправить" а "Получить доступ"
- **Loading State на кнопке** *(GitHub, Stripe)*
- **CTA в состоянии готовности**

### Обратная связь и состояния
- **Optimistic UI** *(Twitter, Notion)*
- **Toast вместо Modal** *(Linear, Figma)*
- **Error State с решением + без стыда** *(Stripe)*
- **Undo вместо подтверждения** *(Gmail, Notion)*
- **После успеха — следующая глава** *(новый)*

### Доверие и конверсия
- **Trust Signals в точке решения**
- **Risk Reversal** — снять страх до действия
- **Social Proof с конкретикой**
- **Прогресс-индикатор**

### Грамматика движения *(Мохой-Надь — новая категория)*
Каждая анимация имеет значение — не только функцию:
- Появление снизу = рост, приоритет
- Растворение = завершение, уход
- Следование за пальцем = контроль, доверие
- Пружинящий отклик = живость, отзывчивость
- Задержка перед результатом = вес, важность

> "Движение должно говорить — не украшать."

### Тактильное качество *(Айв)*
- Точность анимаций — каждое взаимодействие ощущается намеренным
- Haptic feedback как подтверждение
- Sound design как часть голоса продукта
- Микровзаимодействия с весом и характером

### Производительность как UX
- **Perceived Performance**
- **Prefetching** *(Google)*
- **Lazy Loading**
- **Tempo** — намеренное управление скоростью как инструмент

## ФАЗА 11 — SIGNATURE MOMENT
*Одна точка где продукт показывает что у него есть душа.*
*Агент не придумывает — помогает обнаружить.*

**Метод поиска через вопросы:**
1. Есть ли момент который вызывает что-то неожиданное —
   улыбку, удивление, облегчение?
2. Если продукт был человеком — что бы он сделал
   когда пользователь впервые достигает цели?
3. Что в позиции ["верим что X"] можно выразить
   действием интерфейса — не словами?
4. Где пользователь меньше всего ожидает что его удивят?

**Гипотеза агента:**
Одна конкретная гипотеза signature moment
основанная на позиции и партитуре.
Помечена как гипотеза — не решение.

## ФАЗА 12 — МАСШТАБ
*Продукт живёт. Пользователь меняется.*

**Три ступени пользователя:**

| Ступень | Кто | Что меняется |
|---|---|---|
| Новичок | Первые дни | Подсказки, прогресс, поддержка, защита от страха |
| Регулярный | Знает основы | Shortcuts, персонализация, меньше объяснений |
| Эксперт | Power user | Command palette, плотность, скорость, invisibility |

**Между сессиями** *(новый угол)*:
> Что происходит с пользователем когда он НЕ использует продукт?
> Думает ли о нём? Скучает? Рекомендует?
> Лучшие продукты живут в голове между сессиями.
> Как это проектируется — через незавершённость,
> предвкушение, уведомления с ценностью?

**Offboarding** *(новый угол)*:
> Что происходит когда пользователь хочет уйти?
> Как продукт его отпускает?
> Это определяет рекомендации и возврат.
> Великий offboarding — это инвестиция, не провал.

**Незапланированное использование** *(Мираллес)*:
> Как пользователи будут использовать продукт
> способами которые не предусмотрены?
> Дизайн должен это позволять — не запрещать.
> Twitter изобрёл @, # и RT — пользователи.
> Какое пространство для изобретения есть в этом продукте?

**Ловушка масштаба:**
> Что сейчас кажется хорошим решением но станет ограничением
> когда пользователей будет в 10 раз больше?

## ФАЗА 13 — РЕДУКЦИЯ
*Финальный фильтр перед планом.*
*Лучшее решение часто — убрать, а не добавить.*
*(Рамс)*

Агент проходит все предложенные решения через три вопроса:

> **1. Это здесь потому что необходимо —
>    или потому что казалось хорошей идеей?**
>
> **2. Что произойдёт если убрать это?
>    Пользователь потеряет что-то важное —
>    или станет лучше?**
>
> **3. Это решение служит пользователю —
>    или решает внутреннюю потребность команды
>    в полноте и законченности?**

Всё что не прошло фильтр — убирается или откладывается.

## SAAS-БЛОК
*Паттерны специфичные для B2B SaaS — не живут в обычной библиотеке.*
*Агент обращается к этому блоку автоматически если продукт B2B SaaS.*

### Мультитенантность
- **Tenant Context Bar** — всегда видно в контексте какого тенанта работаешь.
  Особенно критично для super admin / impersonation режимов.
- **Tenant Isolation** — данные одного тенанта никогда не видны другому.
  Проверяется на уровне каждого запроса, не только UI.
- **Cross-tenant Navigation** — как переключаться между тенантами
  без потери контекста и без путаницы.

### Роли и права в UI
- **Role-aware Interface** — интерфейс показывает только то что роль может делать.
  Скрытые элементы не создают вопросов "а где кнопка X?"
- **Permission Boundaries** — явные и понятные сообщения когда действие недоступно.
  Не просто "нет доступа" — а почему и к кому обратиться.
- **Privilege Escalation UX** — как пользователь запрашивает расширенный доступ?
  Процесс должен быть очевидным и безопасным.

### Impersonation / Support режим
- **Явный режим "от лица"** — когда admin работает от имени пользователя,
  это визуально очевидно всё время. Нельзя забыть что ты в чужом аккаунте.
- **Audit-прозрачность** — все действия в режиме impersonation логируются.
  Пользователь может видеть кто и когда заходил от его имени.
- **Безопасный выход** — выход из режима impersonation всегда в одном месте,
  всегда очевиден, никогда не требует более одного действия.

### Platform Health
- **System Status в интерфейсе** — администратор видит здоровье системы
  без перехода в отдельные инструменты.
- **Tenant Health Dashboard** — для super admin: состояние всех тенантов,
  аномалии, риски — в одном месте.
- **Degraded Mode UX** — как интерфейс ведёт себя когда часть системы недоступна?
  Явно, без паники, с конкретными действиями.

### Sales Workflow (для SaaS с sales-моделью)
- **Focused Sales Workspace** — менеджер видит только то что нужно для сделки.
  Не CRM общего назначения — пространство под конкретный workflow.
- **Next Action Always Visible** — что делать дальше всегда очевидно.
  Не "посмотри на все задачи" — а "следующий шаг по этому клиенту".
- **Context Preservation** — при переходе между клиентами/сделками
  контекст сохраняется. Не нужно вспоминать где остановился.

### Стандарт состояний UI
*Единый язык состояний по всему продукту — не придумывать заново для каждого экрана.*

| Состояние | Что показываем | Что предлагаем |
|---|---|---|
| Loading | Skeleton или прогресс с контекстом | — |
| Empty | Объяснение + первое действие | CTA к созданию |
| Error | Что пошло не так + адрес проблемы | Retry / Support |
| Partial | Что загрузилось + что ещё грузится | — |
| Success | Что произошло + что дальше | Next action |
| No permission | Почему + к кому обратиться | Request access |

## ФАЗА 15 — RELEASE CHECKLIST
*Добавляется после Фазы 14 — Плана.*
*Перед релизом. Не после.*

### Функциональность
```
□ Главное действие работает end-to-end
□ Все состояния UI покрыты (loading/empty/error/partial/success)
□ Роли и права проверены для каждой роли
□ Мобильная версия проверена на реальном устройстве
□ Критические пути проверены без интернета / на медленном соединении
```

### Визуальная консистентность
```
□ Один primary CTA на каждом экране
□ Семантика цветов/бейджей/иконок единая по всему продукту
□ Dev-артефакты удалены из пользовательского слоя
□ Типографическая шкала соблюдена везде
□ Отступы кратны базовому модулю
```

### Копирайтинг и тон
```
□ Язык сообщений единый по всему продукту
□ Ошибки написаны без стыда и с решением
□ Кнопки используют verb + result, не просто verb
□ Пустые состояния объясняют что делать
□ После успеха — следующий шаг, не тишина
```

### Accessibility
```
□ Contrast ratio 4.5:1 minimum для текста
□ Touch targets 44×44pt minimum
□ Keyboard navigation работает
□ Screen reader проверен на ключевых экранах
```

### Для B2B SaaS дополнительно
```
□ Tenant isolation проверена (данные не утекают между тенантами)
□ Impersonation режим визуально очевиден
□ Audit log пишется для критических действий
□ Role-aware UI проверен для каждой роли
□ Cross-tenant navigation безопасна
```

### Финальный вопрос перед релизом
> "Если завтра этим воспользуется человек который никогда
> не видел продукт — он достигнет главной цели без помощи?"
> Если нет — что именно мешает?
