# Техническая спецификация: Движок синхронизации (Legacy -> Strapi)

**Статус:** Реализовано (Sync Engine 2.0)
**Назначение:** Документация для разработчика/AI-агента, проектирующего архитектуру нового сайта на базе Strapi.

---

## 1. Архитектура системы (Push-модель)

Синхронизация работает по **Push-модели**. 
Текущее Node.js приложение (Worker) выступает в роли активного инициатора:
1. Читает данные из старых источников (MODX MySQL для СПБ, REST API WordPress / БД cisto_ci74 для Челябинска).
2. Трансформирует данные в формат Strapi (Data Mapping).
3. Отправляет данные в Strapi через стандартный REST API (`POST` / `PUT`).

**Strapi выступает исключительно в роли пассивного приемника (API Server).** Никаких плагинов для импорта внутри самого Strapi писать не нужно.

---

## 2. Ключевые механизмы движка (Уже реализованы)

Чтобы избежать падения серверов и дублирования данных при регулярной синхронизации, в движок заложены следующие механизмы:

### 2.1. Умная Дельта-Синхронизация (MD5 Hashing)
Воркер не опирается на даты изменения в старой БД. Вместо этого:
- При формировании JSON-объекта для Strapi, воркер вычисляет его `MD5-хеш`.
- Хеш сохраняется в локальной таблице `sync_map` вместе со связкой ID (`original_id` -> `strapi_id`).
- При следующем запуске воркер снова собирает объект и сверяет хеш. **Если хеш совпадает, запись пропускается (Skip)**. Это экономит 99% запросов к API при регулярных запусках.

### 2.2. Идентификация и предотвращение дублей (State Management)
Локальная таблица `sync_map` хранит состояние:
- Если `original_id` нет в таблице -> выполняется `POST` (Создание), полученный от Strapi `id` сохраняется в базу.
- Если `original_id` есть в таблице -> выполняется `PUT` (Обновление) по сохраненному `strapi_id`.

### 2.3. Батчинг (Batch Processing)
Данные обрабатываются чанками (по умолчанию `chunkSize = 50`). Это предотвращает утечки памяти (Out of Memory) на стороне воркера и защищает Strapi от одновременной обработки тысяч соединений.

### 2.4. Отказоустойчивость (Retry Logic & Exponential Backoff)
Если Strapi возвращает ошибку `429 Too Many Requests` или `5xx Server Error`:
- Запрос не падает, а уходит в ожидание (1 сек, затем 2 сек, затем 4 сек).
- Ошибки валидации (`400 Bad Request`) логируются, счетчик ошибок инкрементируется, но **процесс не останавливается**. Воркер переходит к следующей записи.

## 3. Требования к разработчику Strapi (Action Items)

Для того чтобы этот движок смог успешно пушить данные, на стороне Strapi необходимо обеспечить следующее:

### 3.1. Настройка API-токена
Необходимо создать **API Token** (в панели Strapi: Settings -> API Tokens) с правами `Full Access` или кастомными правами на `find`, `findOne`, `create`, `update` для всех синхронизируемых коллекций (Doctors, Services, Reviews и т.д.), а также правами на `upload` (Media Library).

### 3.2. Структура коллекций (Content Types)
Коллекции в Strapi должны быть спроектированы на основе файла `full_dump.json` (сгенерированного в модуле экспорта). 
*Важно:* Воркер будет мапить старые поля в новые. Если вы называете поле `firstName` вместо `pagetitle`, просто зафиксируйте это в вашей документации по схеме, чтобы мы могли настроить маппер на стороне воркера.

### 3.3. Обработка связей (Relations)
При создании записей со связями (например, Врач -> Услуги), воркер будет передавать массив ID связанных записей в формате Strapi:
\`\`\`json
{
  "data": {
    "name": "Иванов Иван",
    "services": [12, 45, 67] // Это ID записей ВНУТРИ Strapi
  }
}
\`\`\`
*Задача воркера:* Сначала синхронизировать независимые сущности (Услуги, Локации), получить их `strapi_id`, и только потом синхронизировать сущности, которые на них ссылаются (Врачи), подставляя правильные `strapi_id`. От Strapi требуется только стандартная поддержка реляционных полей.

### 3.4. Медиафайлы (Изображения)
**Как это будет работать:**
1. Воркер скачивает картинку со старого сайта.
2. Воркер отправляет картинку в Strapi через `/api/upload` (POST multipart/form-data).
3. Strapi возвращает `id` загруженного файла.
4. Воркер сохраняет хеш картинки и её `strapi_id` в локальную базу.
5. При создании/обновлении карточки Врача, воркер передает `id` картинки в поле `photo`.

*Требование к Strapi:* Стандартный плагин Upload должен быть активен и доступен по API.

### 3.5. Soft Delete (Снятие с публикации)
Мы не будем физически удалять записи через API (`DELETE`). 
Вместо этого, если запись исчезла из старой базы, воркер отправит `PUT` запрос, устанавливая поле `publishedAt: null` (или кастомное поле `isActive: false`).
*Требование к Strapi:* Включить Draft & Publish систему для коллекций, либо предусмотреть boolean поле для статуса активности.

## 4. Пример пайплайна (Как будет выглядеть код маппера)

Для понимания, вот как выглядит функция маппинга, которую мы напишем на стороне воркера после утверждения схемы Strapi:

\`\`\`typescript
// Пример маппера для Врача (Этап 2)
function mapDoctorToStrapi(oldDoc) {
  return {
    title: oldDoc.pagetitle,
    slug: oldDoc.alias,
    experience_years: parseInt(oldDoc.tvs.stazh) || 0,
    description: oldDoc.content,
    // Связи подтягиваются из локальной таблицы sync_map
    locations: oldDoc.tvs.loc.map(locId => getStrapiId('location', locId)),
    // Картинка подтягивается из локальной таблицы media_map
    photo: getStrapiMediaId(oldDoc.tvs.image) 
  };
}
\`\`\`

## Резюме для агента-разработчика Strapi:
1. Изучите `full_dump.json` и `MIGRATION_GUIDE.md`.
2. Спроектируйте Content Types в Strapi.
3. Предоставьте URL сервера Strapi, API Token и итоговую JSON-схему (какие поля получились).
4. Движок синхронизации уже готов взять ваши схемы, написать под них мапперы и бесперебойно залить все данные.
