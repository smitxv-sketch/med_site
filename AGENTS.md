# Проектные правила: Универсальная верстка и Сильная Архитектура (Zero-Maintenance)

## ЧАСТЬ 1: МЕТА-ПАРАДИГМА (Как ИИ должен мыслить)
Перед написанием любого кода, ИИ обязан посмотреть на задачу через призму "Principal Engineer" (Ведущего инженера).
Мы не используем "быстрые/популярные" решения из туториалов, если они вредят производительности или UX. 

**Принципы принятия решений для ИИ:**
1. **Native-First (Нативность):** Если задачу можно решить средствами HTML/CSS (например, `scroll-snap` для слайдеров, `aspect-ratio` для пропорций, `svh` для высоты экрана), строго запрещено использовать для этого JavaScript (стейты, расчеты `window.innerHeight` и сторонние библиотеки).
2. **Zero-Layout-Shift (Отсутствие скачков верстки):** Интерфейс не должен дергаться. Все анимации (появление текста, загрузка картинок) должны происходить внутри жестко зарезервированного пространства (пре-рендер, скелетоны). Анимация должна влиять на `opacity` или `transform`, но не на ширину/высоту (не толкать соседей).
3. **Defensive UI (Защищенный интерфейс):** ИИ должен заранее предполагать худший сценарий: что текста будет в 5 раз больше, что экран будет очень узким, что картинка будет белой (что убьет белый текст на ней).

## ЧАСТЬ 2: UX И ФРОНТЕНД ПРАВИЛА
**ГЛАВНАЯ ПАРАДИГМА:** Мы используем "Content-Driven Layout" (Контент определяет размер). Строго запрещен хардкод фиксированных высот для медиа-контейнеров, карточек и баннеров. Интерфейс должен не ломаться при изменении объема текста и пропорций экрана.

### 1. Запрет на жесткие высоты (`h-[px]`/`h-[vh]`)
- ❌ **НЕ ДЕЛАТЬ:** `<div className="h-[400px]">Контент</div>`. `<div className="h-[100vh]">...</div>` на мобильных часто ломается из-за адресной строки.
- ✅ **ДЕЛАТЬ:** Использовать пропорции: `aspect-[4/5]`, `aspect-video`, или `min-h-[100svh]`. Для текстовых блоков использовать `flex-1`.

### 2. Интеграция Safe Areas (Защита от iOS-челок и системных UI)
- Обязательно использовать `pb-safe` для главного блока контента (отступ `env(safe-area-inset-bottom)`).

### 3. Картинки и Баннеры
- Фото-подложки всегда: `absolute inset-0 w-full h-full object-cover`. Своего размера не имеют, подстраиваются под родителя.

### 4. Защитные градиенты (Defensive gradients)
- При наложении светлого текста на картинки использовать двойную защиту: плавно растущий `bg-gradient-to-t` плюс жесткий белый или черный низ `h-1/4`, чтобы последние строки и кнопки читались на 100%.

### 5. Защита от длинных текстов (Line Clamping)
- В сетках и списках жестко ограничиваем длину текста хелперами `line-clamp-2` или `line-clamp-3`.

### 6. Эффекты и Анимации
- Использовать аппаратное ускорение. Для эффектов вроде печатной машинки не изменять DOM (добавляя по букве), а рендерить всё сразу и управлять `opacity` через CSS/JS для сохранения стабильной верстки (Смотри `GhostTyping.tsx` как эталон).

## ЧАСТЬ 3: БИБЛИОТЕКИ И СТЕК (RADIX + SHADCN)
**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО:** В проекте внедрена гибридная архитектура "Tailwind + Radix UI (Shadcn)".

1. **Никаких самописных сложных UI-компонентов:** Если задача требует создания модального окна, попапа, выпадающего списка (select/dropdown) или аккордеона — **ЗАПРЕЩЕНО** писать их с нуля на `div` + `useState`. ИИ **ОБЯЗАН** использовать примитивы Radix UI (или генерировать их через Shadcn паттерны в `src/components/ui/`).
2. **Директория компонентов:** Сгенерированные компоненты Shadcn (Select, Dialog, Sheet) лежат СТРОГО в папке `src/components/ui`. Изменять их бизнес-логикой запрещено. Наши кастомные компоненты живут в `src/shared/ui` или `src/widgets`.
3. **Функция cn():** Для слияния классов всегда импортируем `import { cn } from '@/lib/utils'` (работает на базе `clsx` + `tailwind-merge`).
4. **Анимации компонентов:** Для анимации Radix-модалок мы используем CSS data-атрибуты (`data-[state=open]:animate-in`) через плагин `tailwindcss-animate`.
5. **Ограничение Framer Motion:** Framer Motion используем **ТОЛЬКО** для микро-хореографии (collapse внутри меню, переходы между роутами). Никогда не используем `motion.div` для outer-контейнеров модальных окон, за это отвечает Radix UI + CSS.

## ЧАСТЬ 4: ГИБРИДНАЯ АРХИТЕКТУРА (HYBRID ROUTING & PAGE ENGINE)
**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО:** В проекте используется гибридный подход к созданию страниц, сочетающий CMS-driven (динамический) и FSD-driven (кастомный) методы.

1. **Динамические страницы (Платформенные):** Лендинги, главная страница, простые информационные разделы собираются из готовых виджетов/блоков через `PageRenderer`. Данные хранятся в формате JSON (имитация CMS).
2. **Кастомные страницы (Специфичные):** Сложные интерфейсы со множеством состояний, фильтрацией или формами (например, `/prices`, модуль онлайн-записи `/booking`, сложные каталоги) создаются как хардкод-страницы в директории `src/pages/` или `src/widgets/` (по методологии FSD).
3. **Единый UI-контур:** Независимо от типа страницы (динамическая или кастомная), они обязаны использовать общую обертку (Layout), единую дизайн-систему (Shadcn/Tailwind) и переиспользовать общие UI-компоненты.
4. **Точки пересечения (Связки):** Платформенные блоки могут ссылаться на кастомные страницы посредством прямых ссылок (например, кнопка с `href="/booking?service_id=123"` в прайсе или виджете), объединяя два мира в бесшовный UX. Не нужно пытаться впихнуть сложную бизнес-логику в универсальный виджет — делегируйте её специализированным страницам (уровня application).
